The Real Problem

Naive Deployment এর ভয়ংকর পরিণতি

Junior engineer প্রথমবার production deploy করে কীভাবে? Server বন্ধ করে, নতুন code দেয়, আবার চালু করে। এই সময়ে কেউ app use করতে পারে না। এটাকে বলে Downtime।

💸
Downtime = টাকা ক্ষতি

Amazon এর ১ ঘণ্টার downtime মানে কোটি কোটি টাকা ক্ষতি। Shopify এর ৫ মিনিটের downtime মানে হাজার হাজার merchants এর business বন্ধ।

🐛
Bug Production এ গেলে

নতুন version deploy করার পর bug পেলে, কীভাবে rollback করবে? আগের version এ কীভাবে ফিরবে দ্রুত?

🌊
Big Bang Release

সব features একসাথে deploy — ১০০% users একসাথে নতুন version পায়। Bug হলে সবাই affected।

😱
Fear of Deployment

Traditional teams প্রতি ৩ মাসে একবার deploy করে — ভয়ে। Modern teams প্রতিদিন ১০০ বার deploy করে — confidently।

🎯 Modern Deployment Goal
Zero downtime, instant rollback capability, gradual traffic shift, ও production confidence — এই চারটা লক্ষ্য পূরণ করতে পারলেই তুমি একজন deployment-confident engineer।
Pattern 1

Blue-Green Deployment — দুটো Environment রাখো

Blue-Green মানে হলো সবসময় দুটো identical production environment রাখো। একটা Live (Blue), একটা Idle (Green)। নতুন version Green এ deploy করো, test করো, তারপর traffic switch করো।

Blue-Green Deployment Flow
✅ Blue-Green এর সুবিধা
Instant rollback: Bug পেলে Load Balancer কে আবার Blue তে point করো — ১ সেকেন্ডে rollback। Zero downtime: Traffic switch instantaneous। Testing: Green এ deploy করে production traffic ছাড়াই test করো।
⚠️ সমস্যা
দুটো full environment মানে দ্বিগুণ cost। Database migration complex হয় (দুটো version same DB share করলে schema compatibility দরকার)।
yaml blue-green-k8s.yaml — Kubernetes Blue-Green 
# Blue deployment — current live version
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: blue
  template:
    metadata:
      labels:
        app: myapp
        version: blue
    spec:
      containers:
      - name: app
        image: myapp:v1.0
---
# Green deployment — new version (deploy করো, traffic নেই এখনো)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: green
  template:
    metadata:
      labels:
        app: myapp
        version: green
    spec:
      containers:
      - name: app
        image: myapp:v2.0
---
# Service — শুধু selector পরিবর্তন করে traffic switch হয়
apiVersion: v1
kind: Service
metadata:
  name: app-service
spec:
  selector:
    app: myapp
    version: blue   # এটা green করলেই switch!
  ports:
  - port: 80
    targetPort: 3000
Pattern 2

Canary Release — ধীরে ধীরে Traffic বাড়াও

Canary deployment এর নাম এসেছে coal miners এর "canary bird" থেকে — খনিতে বিষাক্ত gas আছে কিনা পরীক্ষা করার জন্য canary পাখি পাঠানো হতো। ঠিক তেমনি কিছু users কে নতুন version এ পাঠাও, তারপর দেখো সব ঠিক আছে কিনা।

Canary Deployment — Gradual Traffic Shift
yaml canary-nginx.conf — Nginx Weighted Traffic Split 
# Nginx এ weighted upstream দিয়ে canary করো
upstream app_servers {
  # Stable version — 95% traffic
  server stable-v1:3000 weight=95;

  # Canary version — 5% traffic
  server canary-v2:3000 weight=5;
}

server {
  listen 80;

  location / {
    proxy_pass http://app_servers;
  }
}
yaml canary-k8s.yaml — Kubernetes Canary with Argo Rollouts 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: app-rollout
spec:
  replicas: 10
  strategy:
    canary:
      # Step by step traffic increase
      steps:
      - setWeight: 5      # 5% canary তে
      - pause: {duration: 10m}  # 10 মিনিট monitor
      - setWeight: 25     # 25% canary তে
      - pause: {duration: 30m}  # 30 মিনিট monitor
      - setWeight: 50     # 50% canary তে
      - pause: {duration: 1h}   # 1 ঘণ্টা monitor
      # সব ঠিক থাকলে 100% হয়ে যাবে

      # Auto rollback — error rate 5% এর বেশি হলে
      analysis:
        templates:
        - templateName: error-rate-check
        args:
        - name: canary-service
          value: app-canary
  template:
    spec:
      containers:
      - name: app
        image: myapp:v2.0   # নতুন version
Pattern 3

Rolling Update — একটা একটা করে Update করো

Rolling Update হলো সবচেয়ে সহজ zero-downtime pattern। একটা একটা করে instance update করো। একটা পুরনো instance বন্ধ করো, নতুন version তে start করো, health check pass করলে পরের instance এ যাও।

Rolling Update — Pods একটা একটা করে Update হচ্ছে
yaml rolling-update.yaml — Kubernetes Rolling Update 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-deployment
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1   # সর্বোচ্চ ১টা pod একসাথে down থাকতে পারবে
      maxSurge: 1          # সর্বোচ্চ ১টা extra pod চলতে পারবে update এর সময়
  template:
    spec:
      containers:
      - name: app
        image: myapp:v2.0
        # Health check — এটা pass না করলে পরের pod update হবে না
        readinessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          failureThreshold: 3
কোন version থাকতে পারেRolling Update এর সমস্যা
Update চলাকালীন v1 ও v2 দুটোই live থাকেAPI backward compatible না হলে সমস্যা হবে। v1 আর v2 একই DB use করলে schema change সমস্যা।
Bonus Pattern

Feature Flags — Code Deploy ≠ Feature Enable

Feature Flag (বা Feature Toggle) হলো code এ একটা switch যেটা runtime এ on/off করা যায়। Code deploy করো কিন্তু feature বন্ধ রাখো। যখন ready, flag on করো — deploy লাগে না।

🚀
Dark Launch

Feature code আছে কিন্তু off। Internal team কে চালু করো। ঠিক হলে সবার জন্য on।

🎯
A/B Testing

User এর ৫০% কে Feature A, বাকি ৫০% কে Feature B দেখাও। Analytics দিয়ে winner বেছে নাও।

🌍
Gradual Rollout

Bangladesh এ আগে on করো। Problems নেই? তাহলে সব দেশে on।

⚡
Kill Switch

Production এ bug হলে feature flag off করো — instant। Rollback deploy লাগে না।

javascript feature-flags.js — Simple Implementation 
// Feature flags — runtime এ control করা যায়
const flags = {
  newCheckoutFlow: false,     // off এখন
  aiRecommendations: true,   // on সবার জন্য
  darkMode: false,
};

// User-specific flag — beta users পাবে
function isEnabled(flagName, userId) {
  // LaunchDarkly বা Unleash এর মতো service call করা যায়
  const betaUsers = ['user-001', 'user-002', 'user-003'];

  if (flagName === 'newCheckoutFlow') {
    // শুধু beta users পাবে
    return betaUsers.includes(userId);
  }

  return flags[flagName] ?? false;
}

// Feature এর পাশে নতুন code রাখো
async function handleCheckout(req, res) {
  if (isEnabled('newCheckoutFlow', req.user.id)) {
    // নতুন checkout flow (v2)
    return newCheckoutV2(req, res);
  }

  // পুরনো checkout flow (v1) — এখনো safe
  return oldCheckoutV1(req, res);
}

Assignment

  • Feature Flag Design: একটা food delivery app এ "AI-powered order prediction" feature add হচ্ছে। কীভাবে Feature Flag দিয়ে এটা rollout করবে? লেখো: (ক) কখন internally test করবে (খ) কোন users কে beta দেবে (গ) কীভাবে ধীরে ধীরে ১০০% করবে (ঘ) Bug পেলে কীভাবে rollback করবে।
  • Kubernetes YAML পড়া: উপরে দেওয়া Rolling Update YAML পড়ো এবং explain করো: (ক) readinessProbe কী করে? (খ) যদি নতুন pod এর health check fail করে, তাহলে কী হবে? (গ) maxSurge: 1 এর মানে কী?
  • Cost Analysis: তোমার app এ ১০টা EC2 instance আছে। Blue-Green deployment করলে কত instance লাগবে? Canary করলে? Rolling করলে? Monthly cost comparison করো (t3.small = $0.023/hour ধরো)।

1

দুটো Node.js app বানাও

app-v1: GET / returns {"version": "v1.0", "color": "blue"}. app-v2: returns v2.0, green.

2

Nginx load balancer setup করো

প্রথমে proxy_pass http://blue:3000 — Blue তে সব traffic।

3

Docker Compose তে সব service রাখো

blue, green, nginx — তিনটা service। Green শুরুতে চলবে কিন্তু nginx traffic পাঠাবে না।

4

Traffic switch করো

Nginx config এ blue কে green করো, reload করো — docker exec nginx nginx -s reload।

5

Verify করো

curl localhost — এখন v2.0 response আসছে। Rollback: আবার blue করো।

6

Feature Flag add করো (Bonus)

v2 এর code এ একটা feature flag add করো — env var দিয়ে control করো।

yaml docker-compose.yml — Blue-Green Setup 
version: '3.8'
services:
  # Blue environment (v1)
  blue:
    build:
      context: .
      dockerfile: Dockerfile.v1
    environment:
      - VERSION=v1.0
      - COLOR=blue

  # Green environment (v2)
  green:
    build:
      context: .
      dockerfile: Dockerfile.v2
    environment:
      - VERSION=v2.0
      - COLOR=green
      - NEW_FEATURE_ENABLED=false  # Feature flag!

  # Nginx — traffic controller
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on: [blue, green]
nginx nginx.conf — Switch করার জন্য এক line পরিবর্তন করো 
events {}
http {
  server {
    listen 80;
    location / {
      # Blue-Green switch: blue → green তে change করো
      proxy_pass http://blue:3000;   # ← এটাই switch করো
      proxy_set_header Host $host;
    }
  }
}
🎯 এটা করলে যা শিখবে
হাতে-কলমে দেখবে Blue-Green deployment কীভাবে কাজ করে। একটা line পরিবর্তন করে traffic switch হতে দেখবে। Downtime নেই — curl করতে থাকো, response continuously আসতে থাকবে। এটাই zero-downtime।
