⏱ ৪-৫ ঘণ্টা 🎯 Advanced 📊 Monitoring & Tracing
Why It Matters

Production এ যখন কিছু ভাঙে...

রাত ৩টায় alert আসলো — "Users payment করতে পারছে না।" তুমি কী করবে? কোথায় দেখবে? কোন service fail করছে? Database slow নাকি network issue? এই মুহূর্তে Observability না থাকলে তুমি অন্ধের মতো কাজ করবে।

🎯 Core Concept
Observability মানে একটা system এর ভেতরের state সম্পর্কে শুধু external outputs দেখে বোঝার ক্ষমতা। Monitoring বলে "কী হচ্ছে" — Observability বলে "কেন হচ্ছে।"
😱 Without Observability

Error আসছে। ১০টা microservice আছে। কোনটায় সমস্যা? Log দেখতে গেলে ১০টা server SSH করতে হবে। API কতটা slow? জানা নেই।

😌 With Observability

Dashboard খুলে দেখলাম — payment-service এর DB query latency 5s হয়ে গেছে। Trace দেখলাম কোন query। Logs দেখলাম exact error। ৫ মিনিটে fix।

Section 02

Prometheus — Metrics Collection

Prometheus হলো একটা open-source metrics collection ও alerting system। এটা নিজে থেকে configured endpoints থেকে metrics pull করে এবং time-series database এ store করে।

Prometheus এর ৪টা Metric Type

Typeকী করেExample
Counterশুধু বাড়ে, কমে না। Reset হয় restart এ।http_requests_total, errors_total
Gaugeউপরে-নিচে যেতে পারে।memory_usage_bytes, active_connections
HistogramValues কে buckets এ ভাগ করে। Percentile calculate করা যায়।http_request_duration_seconds
SummaryHistogram এর মতো কিন্তু client-side percentile।rpc_duration_seconds

Prometheus Architecture

📌 Push vs Pull
Prometheus pull করে — নিজে গিয়ে metrics নিয়ে আসে। এর মানে service জানে না কখন scrape হবে। Grafana Cloud, Datadog push করে — service নিজে পাঠায়। Pull এর সুবিধা: service discovery সহজ, single config এ সব।

PromQL — Prometheus Query Language

PromQLcommon queries
# Error rate (percentage)
rate(http_requests_total{status=~"5.."}[5m]) /
rate(http_requests_total[5m]) * 100

# P99 latency (99th percentile)
histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket[5m])
)

# Request rate per service
sum(rate(http_requests_total[1m])) by (service)

# Memory usage above 80%
(node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
  / node_memory_MemTotal_bytes * 100 > 80

# Alert rule example
- alert: HighErrorRate
  expr: error_rate > 5
  for: 2m
  labels:
    severity: critical
  annotations:
    summary: "Error rate {{ $value }}% on {{ $labels.service }}"
Section 03

Grafana — Visualization Layer

Grafana হলো metrics এর জন্য beautiful dashboards। Prometheus থেকে data নিয়ে graphs, charts, alerts — সব visualize করে। Prometheus ছাড়াও InfluxDB, Loki, Elasticsearch থেকেও data নিতে পারে।

Golden Signals — কী Monitor করবে?

📊 Latency

Request কতক্ষণ লাগছে complete হতে? Successful ও failed আলাদা track করো। P50, P95, P99 দেখো।

🚦 Traffic

কত requests per second আসছে? Peak load কখন? Unusual spike আছে কিনা?

❌ Errors

Error rate কত? কোন type of error বেশি? 4xx vs 5xx track করো।

💾 Saturation

System কতটা full? CPU, memory, disk, network কতটা utilize হচ্ছে? Bottleneck কোথায়?

Section 04

Jaeger — Distributed Tracing

Microservices এ একটা request ১০টা service এর মধ্য দিয়ে যেতে পারে। কোনটায় slow? কোনটায় error? Metrics দেখে জানা যায় না — Tracing লাগে।

Jaeger প্রতিটা request এ একটা unique Trace ID দেয়, এবং প্রতিটা service hop কে Span হিসেবে track করে।

Trace এর Anatomy

🎯 Tracing দিয়ে কী বুঝলাম?
Total 485ms। postgres-db span 165ms — এটাই bottleneck। Order service এর DB query slow। Index missing অথবা N+1 query হতে পারে। Trace ছাড়া এটা find করতে কতক্ষণ লাগতো?
🔑 OpenTelemetry
OpenTelemetry (OTel) হলো vendor-neutral standard। Code একবার instrument করো — Jaeger, Zipkin, Datadog, যেকোনো backend এ পাঠাতে পারবে। Industry এর standard হয়ে গেছে।
Section 05

ELK Stack — Centralized Logging

১০০টা server এর logs কোথায় দেখবে? প্রতিটায় SSH করে grep করবে? ELK Stack সব logs একজায়গায় নিয়ে আসে এবং powerful search দেয়।

E — Elasticsearch

Distributed search ও analytics engine। Logs index করে রাখে। Full-text search, aggregations। JSON documents store করে।

L — Logstash

Log processing pipeline। বিভিন্ন source থেকে collect করে, transform করে, Elasticsearch এ পাঠায়।

K — Kibana

Elasticsearch এর visualization tool। Log search, dashboards, anomaly detection। Grafana এর মতো কিন্তু logs এর জন্য।

B — Beats / Filebeat

Lightweight log shipper। প্রতিটা server এ agent চলে, logs Logstash বা Elasticsearch এ পাঠায়।

Prometheus Metrics — Node.js

JavaScriptmetrics.js
const client = require('prom-client');

// Default metrics (CPU, memory, event loop) collect করো
client.collectDefaultMetrics({ prefix: 'myapp_' });

// Custom Counter — HTTP requests count করো
const httpRequests = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status'],
});

// Custom Histogram — Request duration measure করো
const httpDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route', 'status'],
  buckets: [0.001, 0.01, 0.05, 0.1, 0.5, 1, 5],
});

// Middleware — সব request automatically track করো
app.use((req, res, next) => {
  const start = Date.now();

  res.on('finish', () => {
    const labels = { method: req.method, route: req.path, status: res.statusCode };
    httpRequests.inc(labels);
    httpDuration.observe(labels, (Date.now() - start) / 1000);
  });
  next();
});

// /metrics endpoint — Prometheus এখান থেকে scrape করবে
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', client.register.contentType);
  res.send(await client.register.metrics());
});

OpenTelemetry Tracing — Node.js

JavaScripttracing.js
const { NodeSDK } = require('@opentelemetry/sdk-node');
const { JaegerExporter } = require('@opentelemetry/exporter-jaeger');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');

// SDK setup — app start এর আগে এটা run করো
const sdk = new NodeSDK({
  traceExporter: new JaegerExporter({
    endpoint: 'http://jaeger:14268/api/traces',
  }),
  instrumentations: [getNodeAutoInstrumentations()],
  // Express, HTTP, DB — সব automatically instrument হয়!
});

sdk.start();

// Custom span তৈরি করো (manual tracing)
const { trace } = require('@opentelemetry/api');
const tracer = trace.getTracer('order-service');

async function processOrder(orderId) {
  const span = tracer.startSpan('process-order');
  span.setAttributes({ 'order.id': orderId, 'order.type': 'premium' });

  try {
    const result = await db.saveOrder(orderId);
    span.setStatus({ code: SpanStatusCode.OK });
    return result;
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: SpanStatusCode.ERROR });
    throw err;
  } finally {
    span.end();
  }
}

Structured Logging — JSON Format

JavaScriptlogger.js (winston)
const winston = require('winston');

const logger = winston.createLogger({
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()   // JSON format — ELK এ parse করা সহজ
  ),
  transports: [new winston.transports.Console()],
});

// Structured logging — searchable fields সহ
logger.info('Order processed', {
  userId: 'usr_123',
  orderId: 'ord_456',
  amount: 1500,
  duration_ms: 245,
  service: 'order-service',
  traceId: span.getContext().traceId,  // Trace ID link করো!
});

// Output: {"level":"info","message":"Order processed",
//   "userId":"usr_123","orderId":"ord_456",
//   "amount":1500,"timestamp":"2024-01-15T03:42:12Z"}

Question 02 / 10
Prometheus কীভাবে metrics collect করে?
A Services Prometheus এ push করে
B Prometheus নিজে configured endpoints থেকে pull করে (scrape)
C Agent install করতে হয় প্রতিটা service এ
D Network packet inspect করে metrics নেয়
সঠিক উত্তর: B — Prometheus pull-based। নির্দিষ্ট interval এ (default 15s) configured targets এর /metrics endpoint থেকে data নিয়ে আসে। Service শুধু /metrics expose করে, বাকি কাজ Prometheus করে।
Question 03 / 10
Distributed Tracing এ "Span" কী?
A Total request এর সময়
B Error এর description
C Metric এর unit
D একটা single operation এর timing — trace এর ছোট একটা unit, যেমন একটা DB query বা HTTP call
সঠিক উত্তর: D — Span হলো একটা named, timed operation। একটা Trace অনেক Span এ composed। প্রতিটা Span এর start time, duration, ও attributes থাকে। Parent-child relationship এ nested হয়।
Question 04 / 10
ELK Stack এ "L" কী কাজ করে?

C Linkerd — service mesh

সঠিক উত্তর: A — Logstash হলো data processing pipeline। Raw logs কে parse করে structured format এ convert করে। Grok patterns দিয়ে unstructured logs থেকে fields extract করে। Input plugins, filter plugins, output plugins নিয়ে configured।
Question 05 / 10
Google এর "Four Golden Signals" কোনগুলো?
A CPU, Memory, Network, Disk
B Availability, Latency, Throughput, Cost
C Latency, Traffic, Errors, Saturation
D Uptime, Response time, Error count, Cost
সঠিক উত্তর: C — Google এর SRE book এর Four Golden Signals: Latency (request কতক্ষণ লাগে), Traffic (কত requests), Errors (কত fail), Saturation (resources কতটা full)। যেকোনো service monitor করতে এই ৪টা দিয়ে শুরু করো।
Question 06 / 10
OpenTelemetry কেন ব্যবহার করা হয়?
A শুধু Jaeger এর জন্য কাজ করে
B Vendor-neutral instrumentation standard — একবার code করো, যেকোনো backend এ পাঠাও
C Metrics এর জন্য Prometheus এর alternative
D শুধু Python এর জন্য
সঠিক উত্তর: B — OpenTelemetry হলো CNCF এর standard। Code once, export anywhere। আজ Jaeger use করলে, কাল Datadog — code change করতে হবে না। Industry standard হয়ে গেছে।
Question 07 / 10
Prometheus এ Counter vs Gauge এর মূল পার্থক্য কী?
A Counter faster, Gauge slower
B Counter string store করে, Gauge number
C Gauge শুধু বাড়তে পারে, Counter দুদিকে যেতে পারে
D Counter শুধু বাড়তে পারে (reset on restart), Gauge উপরে-নিচে যেতে পারে
সঠিক উত্তর: D — Counter: monotonically increasing। Total requests, total errors — কখনো কমে না। Gauge: current value — CPU usage, active connections — বাড়তে ও কমতে পারে। Wrong type use করলে metrics misleading হয়।
Question 08 / 10
Structured logging কেন গুরুত্বপূর্ণ?
A JSON format এ log করলে ELK automatically fields parse করতে পারে এবং specific fields এ search করা সহজ হয়
B Logs ছোট হয়
C Logs encrypt হয়
D শুধু performance উপকার
সঠিক উত্তর: A — Plain text logs: grep করতে হয়। JSON logs: userId:"usr_123" দিয়ে instantly সব সেই user এর logs পাওয়া যায়। Kibana তে fields automatically appear করে। Alerting করা যায় specific field values এ।
Question 09 / 10
PromQL এ `rate(http_requests_total[5m])` কী return করে?
A Last 5 minutes এর total request count
B Average request duration
C Per-second request rate calculated over the last 5 minutes window
D 5 minutes ago এর snapshot
সঠিক উত্তর: C — rate() function counter এর per-second rate calculate করে। [5m] মানে 5-minute window থেকে average। Raw counter কখনো graph এ directly use করো না — rate() দিয়ে convert করো।

তোমার payment microservice এর latency হঠাৎ বেড়ে গেছে। Metrics দেখে বুঝলে — কিন্তু কোন specific DB query এর কারণে? এই তথ্যের জন্য কোনটা দেখবে?

C Grafana dashboard
D CPU metrics
সঠিক উত্তর: B — Metrics বলে "কী হচ্ছে" — Traces বলে "কোথায় হচ্ছে।" Jaeger এ trace দেখলে ঠিক কোন span slow, কোন DB query কতক্ষণ নিচ্ছে সেটা দেখা যাবে। এটাই Observability এর power।
