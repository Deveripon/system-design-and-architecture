The Problem

Traditional CRUD এর সমস্যা কোথায়?

সাধারণ app এ একটাই database — সেখানে Read ও Write দুটোই হয়। ছোট app এ এটা ঠিক আছে। কিন্তু বড় system এ এটা কেন সমস্যা?

📊
Read vs Write মিসমাচ

Write করতে normalized data ভালো (no duplication)। কিন্তু Read করতে denormalized ভালো (fast query, কম JOIN)। একই schema দিয়ে দুটোই optimize করা কঠিন।

🔒
Lock Contention

যখন একজন user data write করছে, তখন অন্যরা read করতে গেলে lock wait করতে হয়। High traffic এ এটা bottleneck হয়।

📝
Audit Trail নেই

Traditional CRUD এ শুধু current state থাকে। "কে, কখন, কী পরিবর্তন করলো" — এই history থাকে না। কিন্তু banking, healthcare তে এটা must।

🔄
Time Travel অসম্ভব

৩ দিন আগে order এর status কী ছিল? CRUD এ জানা যাবে না। শুধু আজকের state আছে।

💡 Solution: দুটো Pattern
CQRS — Command (Write) এবং Query (Read) কে আলাদা করো। দুটোর জন্য আলাদা model, আলাদা database হতে পারে।

Event Sourcing — Current state store না করে, সব events store করো। State হলো সব events এর sum।
Core Pattern

CQRS — Command Query Responsibility Segregation

CQRS মানে হলো একটা system এর Write side (Command) এবং Read side (Query) আলাদা করা। এটা Bertrand Meyer এর CQS principle থেকে এসেছে: "A method should either change state OR return data, not both."

CQRS Architecture — Command ও Query আলাদা Path

typescript commands.ts — Command Handler Pattern 
// Command — একটা intention, একটা action
interface PlaceOrderCommand {
  type: 'PLACE_ORDER';
  userId: string;
  items: { productId: string; quantity: number }[];
  totalAmount: number;
}

// Command Handler — command receive করে, validate করে, execute করে
class PlaceOrderCommandHandler {
  constructor(
    private orderRepo: OrderRepository,
    private eventBus: EventBus
  ) {}

  async handle(cmd: PlaceOrderCommand): Promise<void> {
    // 1. Validate
    if (cmd.items.length === 0) {
      throw new Error('Order must have at least one item');
    }

    // 2. Create domain object
    const order = new Order({
      id: generateId(),
      userId: cmd.userId,
      items: cmd.items,
      status: 'PENDING',
      totalAmount: cmd.totalAmount,
      createdAt: new Date()
    });

    // 3. Write DB তে save করো
    await this.orderRepo.save(order);

    // 4. Event publish করো — Read side আপডেট হবে
    await this.eventBus.publish({
      type: 'OrderPlaced',
      orderId: order.id,
      userId: cmd.userId,
      totalAmount: cmd.totalAmount,
      timestamp: new Date()
    });
    // Command handler void return করে — no data back!
  }
}

// Query Handler — শুধু read করে, কিছু change করে না
class GetOrderQueryHandler {
  constructor(private readRepo: OrderReadRepository) {}

  async handle(orderId: string): Promise<OrderView> {
    // Read DB থেকে denormalized view return করো
    return this.readRepo.findById(orderId);
    // OrderView তে product names, user info — সব precomputed
  }
}
Event Sourcing

Event Sourcing — State নয়, Events Store করো

Traditional approach এ database তে current state থাকে। Order status "DELIVERED" — কিন্তু কীভাবে এখানে এলো সেটা নেই। Event Sourcing এ প্রতিটা state change একটা Event হিসেবে store হয়। Current state বের করতে সব events replay করো।

🏦 Real Life Analogy: Bank Account
তোমার bank account এর balance এখন ৳৫,০০০। Bank কি শুধু "balance = 5000" store করে? না! তারা store করে সব transactions: +10000 (salary), -3000 (rent), -2000 (shopping)। Balance হলো এই transactions এর sum। এটাই Event Sourcing।

typescript event-sourcing.ts — Order Aggregate 
// Event types — প্রতিটা possible state change
type OrderEvent =
  | { type: 'OrderCreated'; orderId: string; userId: string; items: Item[] }
  | { type: 'PaymentProcessed'; amount: number; transactionId: string }
  | { type: 'OrderShipped'; trackingNumber: string; carrier: string }
  | { type: 'OrderDelivered'; deliveredAt: Date }
  | { type: 'OrderCancelled'; reason: string };

// Order State — events এর sum
interface OrderState {
  id: string;
  userId: string;
  status: 'PENDING' | 'PAID' | 'SHIPPED' | 'DELIVERED' | 'CANCELLED';
  totalAmount: number;
  trackingNumber?: string;
}

// Aggregate — events apply করে state বানায়
function applyEvent(state: OrderState, event: OrderEvent): OrderState {
  switch (event.type) {
    case 'OrderCreated':
      return { ...state, id: event.orderId, userId: event.userId, status: 'PENDING' };

    case 'PaymentProcessed':
      return { ...state, status: 'PAID', totalAmount: event.amount };

    case 'OrderShipped':
      return { ...state, status: 'SHIPPED', trackingNumber: event.trackingNumber };

    case 'OrderDelivered':
      return { ...state, status: 'DELIVERED' };

    case 'OrderCancelled':
      return { ...state, status: 'CANCELLED' };

    default: return state;
  }
}

// Events replay করে current state বের করো
function replayEvents(events: OrderEvent[]): OrderState {
  const initialState: OrderState = { id: '', userId: '', status: 'PENDING', totalAmount: 0 };
  return events.reduce(applyEvent, initialState);
}

// ব্যবহার
const events = await eventStore.getEvents('order-123');
const currentState = replayEvents(events);
// currentState.status === 'SHIPPED' ✓

// Time travel — ৩ ঘণ্টা আগের state দেখো
const pastEvents = events.filter(e => e.timestamp < threeHoursAgo);
const pastState = replayEvents(pastEvents);
// pastState.status === 'PAID' ✓ (ship হয়নি তখনো)
Read Models

Projections — Events থেকে Read Model বানাও

Event Store থেকে সরাসরি query করা slow হতে পারে (অনেক events replay করতে হয়)। Projection হলো events process করে একটা optimized read model তৈরি করা। যেমন: "সব PENDING orders এর list" — এটা একটা projection।

📋
Order List Projection

Orders table তে শুধু latest status রাখো। OrderCreated এলে row insert, OrderShipped এলে status update।

📊
Analytics Projection

Daily sales aggregate করো। OrderPlaced event এলে সেই দিনের total বাড়াও।

🔄
Rebuild করা যায়

Projection corrupt হলে বা নতুন view দরকার হলে — Event Store থেকে শুরু থেকে সব events replay করে নতুন projection বানাও।

🎯
Multiple Views

একই events থেকে ভিন্ন projections। Admin view আলাদা, customer view আলাদা।

typescript projection.ts — Order Summary Projection 
// Projection handler — events শুনে read model আপডেট করে
class OrderSummaryProjection {
  constructor(private db: ReadDatabase) {}

  // Event এলে এই function call হবে
  async on(event: OrderEvent): Promise<void> {
    switch (event.type) {
      case 'OrderCreated':
        // Read DB তে নতুন row insert করো
        await this.db.insert('order_summaries', {
          id: event.orderId,
          userId: event.userId,
          status: 'PENDING',
          itemCount: event.items.length,
          createdAt: event.timestamp
        });
        break;

      case 'PaymentProcessed':
        // Status update করো
        await this.db.update('order_summaries',
          { id: event.orderId },
          { status: 'PAID', totalAmount: event.amount }
        );
        break;

      case 'OrderShipped':
        await this.db.update('order_summaries',
          { id: event.orderId },
          { status: 'SHIPPED', trackingNumber: event.trackingNumber }
        );
        break;
    }
  }
}

// Query করা এখন fast — precomputed table থেকে
async function getPendingOrders(userId: string) {
  // Read DB থেকে simple SQL — কোনো replay নেই
  return db.query(
    'SELECT * FROM order_summaries WHERE userId = ? AND status = ?',
    [userId, 'PENDING']
  );
}

CQRS এ Write DB ও Read DB আলাদা রাখলে কী সুবিধা?
শুধু cost কমে
Security বাড়ে
Write DB কে Write optimize করা যায় (normalized), Read DB কে Read optimize করা যায় (denormalized/cached) — দুটো independently scale হয়

javascript bank-account.js — Starter Code 
// Event Store — simple array, append only
const eventStore = [];

// Event create করো এবং store তে add করো
function appendEvent(event) {
  eventStore.push({ ...event, timestamp: new Date(), seq: eventStore.length + 1 });
}

// State apply করার logic
function applyEvent(state, event) {
  switch (event.type) {
    case 'AccountOpened':
      return { ...state, id: event.accountId, balance: 0, isOpen: true };
    case 'MoneyDeposited':
      return { ...state, balance: state.balance + event.amount };
    case 'MoneyWithdrawn':
      return { ...state, balance: state.balance - event.amount };
    default: return state;
  }
}

// Replay করে current state পাও
function getBalance(upToSeq = Infinity) {
  const events = eventStore.filter(e => e.seq <= upToSeq);
  return events.reduce(applyEvent, { balance: 0, isOpen: false });
}

// Commands
function openAccount(id) {
  appendEvent({ type: 'AccountOpened', accountId: id });
}

function deposit(amount) {
  if (amount <= 0) throw new Error('Amount must be positive');
  appendEvent({ type: 'MoneyDeposited', amount });
}

function withdraw(amount) {
  const { balance } = getBalance();
  if (amount > balance) throw new Error('Insufficient funds');
  appendEvent({ type: 'MoneyWithdrawn', amount });
}

// Test করো
openAccount('acc-001');
deposit(10000);
deposit(5000);
withdraw(3000);
deposit(2000);

console.log('Current balance:', getBalance().balance); // 14000
console.log('After 2 events:', getBalance(2).balance);  // 10000 (time travel!)
