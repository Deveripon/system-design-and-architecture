Foundation

Cloud কেন? On-premise এর সমস্যা কী?

আগে সব company নিজেদের server কিনত। মানে physical মেশিন কিনো, data center বানাও, নিজে manage করো। এটাকে বলে On-premise। কিন্তু এতে সমস্যা অনেক।

💰
Upfront Cost

Server কিনতে লক্ষ লক্ষ টাকা লাগে। আর সেটা idle থাকলেও টাকা যাচ্ছে।

⏰
Slow Provisioning

নতুন server লাগবে? Order করো, deliver হোক, setup করো — সপ্তাহ লাগে।

📉
Over/Under Capacity

Traffic কম হলে waste, বেশি হলে crash। Perfect sizing প্রায় impossible।

🔧
Maintenance Burden

Hardware failure, OS update, security patch — সব নিজেকেই করতে হয়।

☁️ Cloud এর Solution
Cloud হলো "Pay as you go" — শুধু যতটুকু use করো, ততটুকুর জন্য pay করো। AWS, GCP, Azure এরকম হাজারো service দেয় যেগুলো minutes এ setup হয়, seconds এ scale হয়। আজকে Netflix এর ১০০% infrastructure AWS এ চলে।

Infrastructure as Code

Terraform — Infrastructure কোড দিয়ে বানাও

আগে সব manually করতে হতো — AWS console এ click করো, form fill করো, server বানাও। এটা error-prone, slow, এবং reproducible না। Infrastructure as Code (IaC) মানে হলো infrastructure define করো code এ, তারপর সেটা automatically create হবে।

🎯 IaC এর সুবিধা
Version control করো (Git), team এর সাথে share করো, যেকোনো সময় destroy করে আবার create করো। "It works on my machine" problem আর নেই — সবার environment exact same।
Terraform Workflow
terraform main.tf — Basic EC2 + S3 Setup 
# Provider configure করো — কোন cloud এ কাজ করবো
provider "aws" {
  region = "ap-southeast-1"  # Singapore region
}

# S3 Bucket বানাও — static files রাখার জন্য
resource "aws_s3_bucket" "app_assets" {
  bucket = "my-app-assets-2024"

  tags = {
    Name        = "App Assets"
    Environment = "production"
  }
}

# EC2 Instance বানাও — app server
resource "aws_instance" "app_server" {
  ami           = "ami-0c55b159cbfafe1f0"  # Ubuntu 22.04
  instance_type = "t3.micro"               # 2 vCPU, 1GB RAM
  key_name      = "my-key-pair"

  tags = {
    Name = "AppServer"
  }
}

# Security Group — কোন port খোলা থাকবে
resource "aws_security_group" "web_sg" {
  name = "web-security-group"

  # HTTP traffic allow
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # HTTPS traffic allow
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # সব outbound traffic allow
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Output — apply এর পর IP address দেখাবে
output "server_ip" {
  value = aws_instance.app_server.public_ip
}
⚠️ State File সম্পর্কে সাবধান
Terraform একটা terraform.tfstate file রাখে — এই file এই সব infrastructure এর current state track করা হয়। এই file কখনো manually edit করবে না এবং Git এ commit করবে না (sensitive data থাকে)। Remote state হিসেবে S3 + DynamoDB use করো।
terraform backend.tf — Remote State Configuration 
# State file S3 তে রাখো — team collaboration এর জন্য
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "production/terraform.tfstate"
    region         = "ap-southeast-1"
    # DynamoDB দিয়ে locking — দুজন একসাথে apply করলে conflict হবে না
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
  }
}
Serverless Architecture

Serverless — Server ছাড়া Code চালাও

"Serverless" মানে server নেই তা না — মানে হলো তোমাকে server manage করতে হবে না। তুমি শুধু function লেখো, AWS বাকি সব করবে। Scaling, OS update, security patch — সব AWS এর দায়িত্ব।

λ
AWS Lambda

Function হিসেবে code deploy করো। Trigger হলে run করবে, না হলে charge নেই। Max 15 min runtime।

🚪
API Gateway

HTTP request এলে Lambda trigger করো। REST API বানাও বিনা server এ।

⚡
Event-driven

S3 তে file upload হলে, SQS তে message এলে, Schedule এ — যেকোনো event এ trigger।

💰
Cost Efficient

শুধু execution time এর জন্য pay করো। ১০ লাখ request? Free tier এ পড়তে পারে।

Serverless API Flow: API Gateway → Lambda → DynamoDB
javascript handler.js — Lambda Function 
// AWS Lambda handler — প্রতিটা invocation এ এই function call হবে
exports.handler = async (event, context) => {
  const { httpMethod, body, pathParameters } = event;

  // API Gateway থেকে আসা request parse করো
  if (httpMethod === 'GET') {
    const userId = pathParameters?.id;

    // DynamoDB থেকে user data fetch করো
    const user = await getUserFromDynamoDB(userId);

    return {
      statusCode: 200,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(user)
    };
  }

  if (httpMethod === 'POST') {
    const data = JSON.parse(body);

    // Validate করো
    if (!data.name || !data.email) {
      return {
        statusCode: 400,
        body: JSON.stringify({ error: 'name and email required' })
      };
    }

    // DynamoDB তে save করো
    await saveUserToDynamoDB(data);

    return {
      statusCode: 201,
      body: JSON.stringify({ message: 'User created', id: data.id })
    };
  }
};

// Environment variable থেকে config নাও — code এ hardcode করবে না

const AWS_REGION = process.env.AWS_REGION || 'ap-southeast-1';
terraform lambda.tf — Lambda + API Gateway 
# Lambda function deploy করো
resource "aws_lambda_function" "api_handler" {
  filename      = "lambda.zip"
  function_name = "api-handler"
  role          = aws_iam_role.lambda_role.arn
  handler       = "handler.handler"
  runtime       = "nodejs18.x"

  environment {
    variables = {
      DYNAMODB_TABLE = aws_dynamodb_table.users.name
    }
  }

  # Memory এবং timeout configure করো
  memory_size = 256   # MB
  timeout     = 30    # seconds
}

# API Gateway বানাও
resource "aws_apigatewayv2_api" "main" {
  name          = "my-api"
  protocol_type = "HTTP"
}

# Lambda কে API Gateway এর সাথে connect করো
resource "aws_apigatewayv2_integration" "lambda" {
  api_id             = aws_apigatewayv2_api.main.id
  integration_type   = "AWS_PROXY"
  integration_uri    = aws_lambda_function.api_handler.invoke_arn
}

Design Patterns

Cloud Architecture Patterns

🎯 Well-Architected Framework
AWS এর Well-Architected Framework হলো best practices এর collection। এর ৫টা pillar: Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization। Senior engineer interview তে এটা জানা must।
🌍
Multi-Region Deployment

একাধিক AWS region এ deploy করো। একটা region down হলে অন্যটা serve করবে। Netflix এর ৩+ region আছে।

📦
S3 + CloudFront CDN

Static files (images, JS, CSS) S3 তে রাখো, CloudFront দিয়ে globally cache করো। Latency ৫x কমে।

🔄
Auto Scaling Group

Traffic বাড়লে automatically নতুন EC2 add হবে, কমলে terminate হবে। Cost এবং performance balance।

🗃️
RDS Read Replicas

Read-heavy workload এর জন্য replica বানাও। Write primary তে, read replica থেকে — performance বাড়ে।

terraform autoscaling.tf — Auto Scaling Group 
# Launch Template — প্রতিটা নতুন EC2 এর config
resource "aws_launch_template" "app" {
  name_prefix   = "app-"
  image_id      = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.small"

  # Startup script — instance ready হলে app install হবে
  user_data = base64encode(<<-EOF
    #!/bin/bash
    apt-get update
    apt-get install -y nodejs npm
    git clone https://github.com/myapp/api /app
    cd /app && npm install
    node server.js &
  EOF)
}

# Auto Scaling Group — minimum 2, maximum 10 instances
resource "aws_autoscaling_group" "app" {
  desired_capacity = 2
  min_size         = 2
  max_size         = 10

  launch_template {
    id      = aws_launch_template.app.id
    version = "$Latest"
  }

  # CPU ৭০% এর বেশি হলে scale out করো
  tag {
    key                 = "Name"
    value               = "AppServer"
    propagate_at_launch = true
  }
}

# Scaling Policy — CPU based
resource "aws_autoscaling_policy" "cpu_policy" {
  name                   = "cpu-scale-out"
  policy_type            = "TargetTrackingScaling"
  autoscaling_group_name = aws_autoscaling_group.app.name

  target_tracking_configuration {
    predefined_metric_specification {
      predefined_metric_type = "ASGAverageCPUUtilization"
    }
    target_value = 70.0  # CPU 70% হলে scale
  }
}

Q3 / 10
AWS Lambda এর "Cold Start" বলতে কী বোঝায়?
প্রথমবার বা idle থাকার পর invoke হলে container initialize হতে সময় লাগে
Lambda function এ bug আছে
Database connection fail হয়েছে
Region এ network problem আছে
✅ Lambda "Warm" থাকলে (recently invoked হলে) milliseconds এ response দেয়। কিন্তু "Cold" হলে (নতুন container spin up করতে হলে) ১০০ms থেকে কয়েক seconds লাগতে পারে। Python/Node.js এর cold start কম, Java এর বেশি।
Q4 / 10
VPC এর Private Subnet এ রাখা resource internet থেকে directly access করা যায় না — কারণ কী?
Private Subnet এ IP address থাকে না
Firewall block করে রাখে সবসময়
AWS automatically সব port বন্ধ করে
Private Subnet এর Route Table এ Internet Gateway এর route নেই
✅ Internet access হয় Route Table এর মাধ্যমে। Public Subnet এর Route Table এ 0.0.0.0/0 → Internet Gateway route থাকে। Private Subnet এ এই route নেই, তাই internet থেকে directly access হয় না। Database এই জন্যই private subnet এ রাখা হয়।
Q5 / 10
AWS S3 কোন ধরনের storage?

✅ tfstate file এ database passwords, API keys, private IP addresses থাকে। Git এ গেলে সবাই দেখতে পারবে। আর দুজন একসাথে apply করলে state corrupted হতে পারে। Solution হলো S3 remote state + DynamoDB locking।

✅ Auto Scaling Group CloudWatch metrics monitor করে। CPU ৭০% এর বেশি হলে Launch Template অনুযায়ী নতুন instance তৈরি করে Load Balancer এ add করে। Traffic কমলে instance terminate করে cost save করে। এটাই horizontal auto-scaling।
