Security:
JWT, OAuth2, mTLS & Zero Trust

⏱ ৪-৫ ঘণ্টা 🎯 Advanced 🔐 Security Architecture
Why It Matters

Security কেন এত গুরুত্বপূর্ণ?

Software engineer হিসেবে তুমি feature build করতে পারো, database design করতে পারো — কিন্তু যদি system secure না হয়, সব কিছুই বৃথা। ২০২৩ সালে বিশ্বের সবচেয়ে বড় data breach গুলোর ৮০% এর কারণ ছিল authentication বা authorization এর weakness।

🎯 Core Concept
Security এর তিনটা মূল প্রশ্ন: Authentication — "তুমি কে?" | Authorization — "তুমি কী করতে পারো?" | Trust — "আমি তোমাকে কতটুকু বিশ্বাস করবো?"
🏦 Banking App

User login করে, token পায়, সেই token দিয়ে তার account access করে। অন্য কেউ সেই token পেলে সব টাকা নিয়ে যেতে পারে।

🏥 Healthcare System

Doctor patient record দেখতে পারবে, কিন্তু billing department সব record দেখতে পারবে না। Role-based access critical।

🛒 E-commerce

"Google দিয়ে login করো" — OAuth2 দিয়ে Google তোমার identity verify করে, password তোমার কাছেই থাকে।

🔗 Microservices

Service A যখন Service B কে call করে, B কীভাবে জানবে যে এটা A নাকি কোনো attacker? এখানে mTLS লাগে।

Section 02

OAuth2 — Authorization Framework

OAuth2 হলো একটা authorization framework যেটা third-party application কে user এর resources access করার permission দেয় — user এর password share না করে।

তুমি যখন "Login with Google" ক্লিক করো — এটাই OAuth2। তোমার Google password কিন্তু ওই app পায় না। Google শুধু confirm করে "হ্যাঁ এই লোক আমার কাছে registered"।

OAuth2 এর ৪টা Role

👤 Resource Owner

তুমি — যে user। তোমার data কার কাছে যাবে সেটা তুমি approve করো।

📱 Client App

Third-party app (যেমন Spotify) যে তোমার Google profile দেখতে চায়।

🔐 Authorization Server

Google, Facebook — যে আসলে token issue করে।

📦 Resource Server

Google এর API — যেখানে তোমার actual data আছে।

Authorization Code Flow (সবচেয়ে secure)

🔑 কেন Authorization Code?
Code টা browser এ যায়, কিন্তু actual access token browser এ আসে না — server-to-server exchange হয়। তাই browser history, logs এ token expose হয় না। এটাই Authorization Code Flow এর security advantage।

Section 03

mTLS — Mutual TLS

Normal HTTPS (TLS) এ শুধু server নিজেকে prove করে certificate দিয়ে। Client verify করে "হ্যাঁ এটা সত্যিই google.com"। কিন্তু server জানে না client কে।

mTLS (Mutual TLS) এ দুই দিকই certificate দেয়। Server ও client — দুজনেই নিজেদের identity prove করে। এটা microservices communication এর জন্য perfect।

🔑 Certificate Hierarchy
CA (Certificate Authority) — সবার উপরে, সব cert sign করে।
Intermediate CA — Team বা department level।
Leaf Certificate — Individual service এর cert।

Service Mesh যেমন Istio বা Linkerd automatically প্রতিটা pod এর জন্য cert issue ও rotate করে।
Section 04

Zero Trust Architecture

Traditional security: "Trust but verify" — corporate network এর ভেতরে থাকলে বিশ্বাস করো।

Zero Trust: "Never trust, always verify" — network এর ভেতরে থাকলেও বিশ্বাস করো না। প্রতিটা request আলাদাভাবে authenticate ও authorize করো।

💀 কেন Traditional Perimeter Security ফেল করে?
VPN দিয়ে once inside হলে সব trust। কিন্তু একজন employee এর laptop hack হলে? সে তো already "trusted network" এ। তখন attacker পুরো internal network এ ঘুরে বেড়াতে পারে। এটাই lateral movement attack।

Zero Trust এর ৫টা Principle

1️⃣ Verify Explicitly

সবসময় authenticate ও authorize করো। User identity, device health, location, service — সব check করো। একবার logged in মানেই trusted না।

2️⃣ Least Privilege Access

ঠিক যতটুকু permission দরকার, ততটুকুই দাও। Admin থাকলেই সব পাবে না — task-specific access দাও। JIT (Just-in-Time) access এর মাধ্যমে।

3️⃣ Assume Breach

ধরে নাও যে breach হয়েছে বা হবে। Blast radius minimize করো। Micro-segmentation করো — একটা service compromise হলে বাকিরা safe থাকে।

4️⃣ Continuous Monitoring

সব activity log করো। Anomaly detect করো। User সকালে Bangladesh থেকে login করে, তারপর ৫ মিনিটে Russia থেকে? Block করো।

5️⃣ Device Trust

শুধু User identity না — device ও trusted কিনা check করো। Unpatched OS? Personal phone? Unknown device? Conditional access apply করো।

JWT — Node.js Implementation

JavaScriptjwt-auth.js
const jwt = require('jsonwebtoken');

// ১. Token তৈরি করো (Login এর সময়)
function generateToken(user) {
  const payload = {
    sub: user.id,          // Subject (user ID)
    email: user.email,
    role: user.role,
    iat: Math.floor(Date.now() / 1000),   // Issued At
    exp: Math.floor(Date.now() / 1000) + (60 * 60), // 1 hour expiry
  };

  return jwt.sign(payload, process.env.JWT_SECRET, {
    algorithm: 'HS256'
  });
}

// ২. Token verify করো (প্রতিটা protected route এ)
function verifyToken(token) {
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    return { valid: true, payload: decoded };
  } catch (err) {
    return { valid: false, error: err.message };
  }
}

// ৩. Express Middleware
function authMiddleware(req, res, next) {
  const authHeader = req.headers['authorization'];
  const token = authHeader?.startsWith('Bearer ')
    ? authHeader.slice(7) : null;

  if (!token) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const result = verifyToken(token);
  if (!result.valid) {
    return res.status(401).json({ error: 'Invalid token' });
  }

  req.user = result.payload;  // Next middleware এ user পাওয়া যাবে
  next();
}

// ৪. Role-based Authorization
function requireRole(role) {
  return (req, res, next) => {
    if (req.user?.role !== role) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

// Usage:
app.get('/admin/users', authMiddleware, requireRole('admin'), handler);

OAuth2 — Google Login (Express)

JavaScriptoauth2-google.js
const { OAuth2Client } = require('google-auth-library');
const client = new OAuth2Client(process.env.GOOGLE_CLIENT_ID);

// ১. Frontend থেকে Google token আসে, সেটা verify করো
app.post('/auth/google', async (req, res) => {
  const { idToken } = req.body;

  try {
    // Google এর কাছে token verify করো
    const ticket = await client.verifyIdToken({
      idToken,
      audience: process.env.GOOGLE_CLIENT_ID,
    });

    const { email, name, picture, sub: googleId } = ticket.getPayload();

    // User কে DB তে খোঁজো বা নতুন create করো
    let user = await User.findOne({ email });
    if (!user) {
      user = await User.create({ email, name, picture, googleId });
    }

    // আমাদের নিজস্ব JWT issue করো
    const token = generateToken(user);
    res.json({ token, user: { email, name, picture } });

  } catch (err) {
    res.status(401).json({ error: 'Invalid Google token' });
  }
});

mTLS — Python গ্রপিসি উদাহরণ

Pythonmtls_server.py
import ssl
import grpc

# mTLS সহ gRPC server
def create_mtls_server():
    # Server এর নিজের certificate
    server_cert = open('server.crt', 'rb').read()
    server_key  = open('server.key', 'rb').read()

    # CA certificate (client cert এর issuer)
    ca_cert = open('ca.crt', 'rb').read()

    # Credentials তৈরি — client cert REQUIRED
    credentials = grpc.ssl_server_credentials(
        [(server_key, server_cert)],
        root_certificates=ca_cert,
        require_client_auth=True,  # মূল পার্ট!
    )

    server = grpc.server(...)
    server.add_secure_port('[::]:50051', credentials)
    return server

# Client side
def create_mtls_channel():
    client_cert = open('client.crt', 'rb').read()
    client_key  = open('client.key', 'rb').read()
    ca_cert     = open('ca.crt', 'rb').read()

    credentials = grpc.ssl_channel_credentials(
        root_certificates=ca_cert,
        private_key=client_key,
        certificate_chain=client_cert,
    )
    return grpc.secure_channel('service-b:50051', credentials)

JWT এর Payload অংশে কী থাকে এবং এটা কি secure?
A Encrypted data — কেউ পড়তে পারবে না
B Base64 encoded claims — decode করলে সবাই পড়তে পারবে, কিন্তু modify করলে signature invalid হয়
C Hashed data — one-way, কেউ পড়তে পারবে না
D Plain text — কোনো encoding নেই
সঠিক উত্তর: B — JWT এর payload শুধু Base64URL encoded, encrypted না। যে কেউ decode করে পড়তে পারবে। তাই password, credit card নম্বর payload এ রাখা উচিত না। কিন্তু signature থাকায় modify করলে verification fail করবে। Confidentiality চাইলে JWE (JSON Web Encryption) ব্যবহার করতে হবে।

OAuth2 এ "Authorization Code Flow" এ Code টা কোথায় যায় এবং কেন এটা সবচেয়ে secure?
A সরাসরি browser এ access token আসে
B Code browser এ যায়, browser থেকে সরাসরি API call করে token নেয়
C Code browser এ যায়, কিন্তু actual token exchange server-to-server হয় (client secret সহ) — browser কখনো token দেখে না

সঠিক উত্তর: C — Authorization Code browser এ redirect query param হিসেবে আসে। কিন্তু backend server সেই code + client_secret দিয়ে auth server থেকে access token নেয়। Client secret browser এ কখনো expose হয় না। Token ও browser history তে থাকে না। এটাই security advantage।
Question 03 / 10
mTLS এবং regular TLS এর মূল পার্থক্য কী?
A mTLS আরো strong encryption algorithm ব্যবহার করে
B mTLS শুধু HTTP/2 তে কাজ করে
C mTLS এ certificate লাগে না, username/password লাগে
D Regular TLS এ শুধু server certificate verify হয়, mTLS এ client ও server উভয়ই certificate দিয়ে নিজেকে verify করে
সঠিক উত্তর: D — TLS = Server proves its identity to client। mTLS = Server AND Client দুজনেই certificate দিয়ে identity prove করে। Microservices এ এটা critical কারণ Service A নিশ্চিত করতে চায় যে সে Service B এর সাথেই কথা বলছে, কোনো attacker এর সাথে না।

সঠিক উত্তর: B — Algorithm "none" মানে কোনো signature নেই। Attacker payload এ role: "admin" দিয়ে token তৈরি করতে পারে, এবং server সেটা valid মনে করবে। এটা একটা বিখ্যাত JWT vulnerability। Library গুলো explicitly এটা reject করে কিন্তু misconfiguration এ bypass সম্ভব।

Zero Trust এ "Lateral Movement" attack কী এবং এটা কীভাবে prevent করা হয়?
A Network এর এক part compromise হলে attacker অন্য parts এ যায় — Micro-segmentation দিয়ে prevent করা হয়
B Attacker horizontally scale করে
C Database এ side-channel attack
D DNS spoofing attack
সঠিক উত্তর: A — Lateral movement: একটা server compromise করার পর attacker সেখান থেকে internal network এ ঘুরে বেড়ায় এবং sensitive systems এ access করে। Zero Trust micro-segmentation দিয়ে prevent করে — প্রতিটা service এর blast radius limit করা হয়।
Question 10 / 10
তুমি একটা fintech startup এর backend design করছো যেখানে microservices আছে। Service-to-service communication secure করতে কোন approach নেবে?
A প্রতিটা service এ username/password authentication রাখো
B Services গুলো same network এ থাকায় কিছু করতে হবে না
C Service mesh (Istio/Linkerd) দিয়ে mTLS enable করো — automatic cert rotation ও zero-trust by default
D শুধু API gateway level এ JWT verify করলেই হবে
সঠিক উত্তর: C — Fintech এ service-to-service security critical। Service mesh + mTLS = automatic identity, encryption, ও zero trust সব একসাথে। API gateway JWT শুধু external traffic handle করে। Internal service communication আলাদাভাবে secure করতে হবে। Same network এ থাকা কোনো protection না।

🔐 Secure Todo API

1

Project Setup

npm init -y করো। Install: express jsonwebtoken bcryptjs। Basic Express server চালু করো।

2

User Registration & Login

POST /auth/register — password bcrypt দিয়ে hash করো। POST /auth/login — credentials verify করে JWT return করো।

3

JWT Middleware

Authorization header থেকে token extract করো। Verify করো। req.user এ decoded payload রাখো।

4

Role-based Routes

GET /todos — authenticated user এর todos। GET /admin/users — শুধু admin role পাবে। অন্যরা 403 পাবে।

5

Test with Postman

Register করো → Login করো → Token নাও → Protected route call করো। Invalid token দিয়ে test করো। Admin route non-admin দিয়ে test করো।

6

Bonus: Refresh Token

Access token (15min) + Refresh token (7 days) implement করো। Refresh token rotate করো প্রতিবার use এ।
