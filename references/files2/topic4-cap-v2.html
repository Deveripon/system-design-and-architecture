Consistency

⏱ ৬০-৯০ মিনিট 📊 Junior Level 🏁 Phase 1 Final
Why It Matters

CAP Theorem কেন শিখতে হবে?

তুমি একটা distributed system design করছো। Interview এ জিজ্ঞেস করলো — "তোমার system এ network failure হলে কী হবে? User কি wrong data দেখবে নাকি error পাবে?" — এই question এর answer হলো CAP Theorem।

Cassandra vs MongoDB vs PostgreSQL কোনটা choose করবে — এই decision এর পেছনে CAP আছে। Senior engineer হতে হলে এই trade-off বুঝতেই হবে।

📌 Definition
CAP Theorem বলে: একটা distributed system এ সর্বোচ্চ দুটো guarantee দেওয়া সম্ভব — Consistency, Availability, Partition Tolerance এর মধ্যে। তিনটা একসাথে impossible। Eric Brewer 2000 সালে propose করেন, 2002 সালে formally prove হয়।
CAP Theorem

C, A, P — তিনটা কি মানে?

C — Consistency

সব nodes এ same time এ same data। Write করার পর যেকোনো node থেকে read করলে latest value পাওয়া যাবে। "All nodes see the same data simultaneously।"

A — Availability

প্রতিটি request এর response আসবেই — error নয়, timeout নয়। কিছু nodes down থাকলেও system respond করবে। তবে response টা stale হতে পারে।

P — Partition Tolerance

Network partition (nodes এর communication বিচ্ছিন্ন) হলেও system কাজ করতে থাকে। Distributed system এ P সবসময় দরকার — network কখনো ১০০% reliable না।

💡 Real Choice

Distributed system এ P বাদ দেওয়া যায় না। তাই real choice হলো: CP নাকি AP? Network partition হলে কোনটা sacrifice করবো — Consistency নাকি Availability?

🎯 Key Insight — Interview এ বলো
"CAP এ P সবসময় থাকবে কারণ network failure real world এ inevitable। তাই practical choice হলো CP বা AP। Financial systems এ CP — wrong data catastrophic। Social media তে AP — stale data কয়েক seconds acceptable।"
CP vs AP

Network Partition হলে কি হয়?

CP Systems — Consistency Priority

Partition হলে কিছু requests reject করে। Wrong data দেবে না। Banking, coordination (Zookeeper), configuration management এর জন্য।

AP Systems — Availability Priority

Partition হলেও respond করে, কিন্তু stale data দিতে পারে। Eventually consistent। Social feeds, caching, DNS, shopping cart এর জন্য।

Consistency Levels

Consistency — Strong থেকে Eventual

Consistency binary না — একটা spectrum। কতটা strong consistency চাই সেটা per-use-case decide করতে হয়।

Levelমানে কিReal Example
StrongWrite এর পর সব nodes এ latest value। SlowestBank balance, Stock inventory
LinearizableReal-time order maintain। Operations globally orderedZookeeper, etcd
CausalCausally related operations ordered। Reply এর আগে post দেখা যাবেChat apps, Comments
Read-your-writesতুমি নিজের write সবসময় দেখবে। অন্যরা হয়তো দেখবে নাInstagram — নিজের post দেখা
EventualEventually সব nodes same data। কখন হবে guarantee নেইDNS, YouTube view count, Likes
💡 Eventual Consistency — বাস্তব উদাহরণ
DNS update করলে সারা বিশ্বে propagate হতে ২৪-৪৮ ঘণ্টা লাগতে পারে। এই সময় কেউ old IP দেখছে, কেউ new IP। এটাই Eventual Consistency। Facebook likes, YouTube view count, Twitter follower count — সবই eventually consistent।

-- ALL: সব nodes confirm করতে হবে (strongest, slowest)
UPDATE accounts SET balance = balance - 5000
WHERE account_id = 101
CONSISTENCY ALL;            -- Critical financial op
PACELC Theorem

PACELC — CAP এর Extension

CAP শুধু partition সময়ের কথা বলে। কিন্তু normal operation এও trade-off আছে। Daniel Abadi 2012 সালে PACELC propose করেন।

📌 PACELC মানে
Partition থাকলে Availability vs Consistency; Else (normal এ) Latency vs Consistency। যত বেশি nodes এ replicate করবে, তত বেশি consistent কিন্তু তত বেশি latency।
DatabasePartition TimeNormal Operationসহজ কথায়
DynamoDBPA (available)EL (low latency)Fast, eventual consistency
CassandraPA (available)EL (configurable)Tunable, default fast
MongoDBPC (consistent)EC (some latency)Strong consistency
PostgreSQLPC (consistent)EC (ACID overhead)Full ACID, slower
Real World Examples

বড় কোম্পানিগুলো কীভাবে করেছে

📦 Amazon DynamoDB — AP by Design

Amazon consciously AP choose করেছে। "Always available" guarantee। Werner Vogels (Amazon CTO): "Network partition হলে তোমাকে পুরোনো cart দেখাবো কিন্তু cart lock করবো না।" Availability থেকে Consistency বেশি important — cart lock হলে sale হারাবে। তবে checkout এ CP enforce হয়।

🔒 Apache Zookeeper — CP by Design

Distributed coordination service — CP choice। Leader election, distributed locks, config management। Wrong value এখানে catastrophic — দুটো nodes যদি একসাথে leader মনে করে, split-brain problem হয়। Partition এ কিছু clients reject হলেও correct data দেওয়া জরুরি। Kafka broker coordination, HDFS এ Zookeeper use হয়।

💡 Split-brain কি?
Network partition এ দুটো partition নিজেকেই primary মনে করে — conflicting states তৈরি হয়। Solution: Quorum (majority voting) — ছোট partition writes accept করে না। Zookeeper, etcd এভাবে handle করে।
Pythoneventual-consistency-sim.py
import asyncio, time, random

class Node:
    def __init__(self, node_id):
        self.node_id = node_id
        self.data = {}       # Local state
        self.peers = []

    async def write(self, key, value):
        # AP: Local write, async sync — fire and forget
        self.data[key] = {'value': value, 'ts': time.time()}
        print(f"[{self.node_id}] Wrote {key}={value} (local)")
        asyncio.create_task(self._propagate(key))  # Async!

    async def read(self, key):
        # May return stale data!
        entry = self.data.get(key)
        val = entry['value'] if entry else None
        print(f"[{self.node_id}] Read {key}={val}")
        return val

    async def _propagate(self, key):
        # Simulate network delay (0.5-3 seconds)
        await asyncio.sleep(random.uniform(0.5, 3.0))
        for peer in self.peers:
            await peer._sync(key, self.data[key])

    async def _sync(self, key, entry):
        # Last-Write-Wins conflict resolution
        existing = self.data.get(key)
        if not existing or existing['ts'] < entry['ts']:
            self.data[key] = entry
            print(f"[{self.node_id}] Synced {key}={entry['value']}")

async def demo():
    a, b = Node("A"), Node("B")
    a.peers = [b]; b.peers = [a]

    await a.write("likes", 100)
    await b.read("likes")    # None! Not synced yet
    await asyncio.sleep(4)  # Wait for propagation
    await b.read("likes")    # 100 — eventually consistent!

asyncio.run(demo())

Bank account balance update এর জন্য কোন consistency model choose করবে?

Question 04 / 10

Question 05 / 10

সঠিক উত্তর: C — CAP শুধু partition সময়। PACELC বলে partition না হলেও (Else) Latency vs Consistency trade-off থাকে। বেশি replication = বেশি consistency কিন্তু বেশি latency।
Question 06 / 10
Split-brain problem কি?
A Database index corruption
B Partition এ দুটো nodes নিজেকে primary মনে করে conflicting state তৈরি করে
C Memory overflow এ process crash
D CPU bottleneck এ slow queries
সঠিক উত্তর: B — Split-brain: partition হলে দুটো side নিজেকে leader মনে করে আলাদাভাবে write accept করে। Partition heal হলে conflicting state থাকে। Quorum voting দিয়ে prevent করা হয় — minority side writes reject করে।
Question 07 / 10
Cassandra তে QUORUM consistency মানে কি?
A শুধু একটা node respond করলেই হবে
B সব nodes respond করতে হবে
C যেকোনো দুটো node respond করলেই হবে
D Majority nodes (N/2 + 1) respond করতে হবে
সঠিক উত্তর: D — QUORUM = floor(N/2) + 1। 3 nodes এ QUORUM = 2। 5 nodes এ QUORUM = 3। একটা node down থাকলেও কাজ করে কিন্তু majority এর agreement নেওয়া হয় — balance between availability and consistency।

সঠিক উত্তর: A — তুমি post করলে নিজের feed এ সাথে সাথে দেখবে। কিন্তু friends হয়তো কয়েক seconds পরে দেখবে। Instagram, Twitter এই model follow করে। Sticky session বা Primary read দিয়ে implement করা হয়।

Node.jsdistributed-node.js
const express = require('express');
const http = require('http');

const PORT = process.env.PORT || 3001;
const PEERS = (process.env.PEERS || '').split(',').filter(Boolean);

const store = new Map();  // In-memory KV store
const app = express();
app.use(express.json());

async function replicateToPeers(key, entry, needed) {
  let acks = 0;
  for (const peer of PEERS) {
    try {
      await fetch(`http://${peer}/replicate`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ key, entry })
      });
      acks++;
    } catch (e) { /* peer unreachable */ }
  }
  return acks;
}

app.put('/:key', async (req, res) => {
  const { key } = req.params;
  const consistency = req.query.consistency || 'QUORUM';
  const entry = { value: req.body.value, ts: Date.now() };

  store.set(key, entry);  // Local write

  const needed = { ONE: 0, QUORUM: 1, ALL: PEERS.length }[consistency];
  const acks = await replicateToPeers(key, entry, needed);

  if (acks < needed) return res.status(503).json({ error: 'Insufficient acks' });
  res.json({ ok: true, acks: acks + 1 });
});

app.listen(PORT, () => console.log(`Node on :${PORT}`));

ONE, QUORUM, ALL consistency level এ আসলে কতটা latency difference হয় সেটা নিজে measure করবে। Partition simulate করলে CP mode কেন error দেয় সেটা বুঝবে। এটাই CAP Theorem এর heart।
