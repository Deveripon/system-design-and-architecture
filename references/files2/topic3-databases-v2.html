Why It Matters

Database কেন এত গুরুত্বপূর্ণ?

একটা system এর performance এর ৮০% নির্ভর করে database design এর উপর। ভুল database choose করলে, index না দিলে, বা wrong schema করলে — ১০০ জন user এলেই সব crash করে।

Interview এ "Design Instagram" বললে — কোন database? SQL নাকি NoSQL? Photos কোথায়? Follow relationship কীভাবে? Feed কীভাবে fast? — সব database question। Database choice ই system এর সবচেয়ে critical decision।

📌 Definition
Database হলো organized data storage system। শুধু store নয় — efficiently retrieve, update, delete এবং manage করাই database এর কাজ। একই query milliseconds এ return করবে নাকি seconds এ — সেটা নির্ভর করে database design এর উপর।
SQL vs NoSQL

SQL vs NoSQL — সবচেয়ে বড় decision

প্রতিটি project এর শুরুতে এই decision নিতে হয়। কোনো একটা "সবসময় better" না — use case বুঝে decide করতে হয়।

📊 SQL — Relational

Structured data, ACID transactions, complex JOINs। Fixed schema। Banking, e-commerce orders, user accounts এর জন্য best।

🌊 NoSQL — Non-Relational

Flexible schema, horizontal scaling, high throughput। Social feed, caching, IoT, product catalog এর জন্য best।

🎯 Interview এ এটা বলো
"আমি SQL দিয়ে শুরু করবো কারণ ACID guarantee এবং mature tooling। Data model unclear বা massive write scale দরকার হলে NoSQL consider করবো। Real production systems এ সাধারণত দুটোই use হয় — Polyglot Persistence।" এটাই industry standard answer।
ACID Properties

ACID — Transaction এর ৪টি Guarantee

ACID মানে Atomicity, Consistency, Isolation, Durability। Bank transfer ভাবো — টাকা একটা account থেকে কাটলে অবশ্যই অন্যটায় যোগ হতে হবে। এই guarantee দেয় ACID।

SQLbank-transfer.sql
-- Bank Transfer — ACID transaction এর classic example
BEGIN TRANSACTION;

-- Step 1: Sender থেকে টাকা কাটো
UPDATE accounts SET balance = balance - 5000
WHERE account_id = 101;

-- Step 2: Balance negative হলে সব undo — Atomicity!
IF (SELECT balance FROM accounts WHERE account_id = 101) < 0 THEN
    ROLLBACK;  -- দুটো UPDATE এর কোনোটাই হবে না
    RAISE EXCEPTION 'Insufficient balance';
END IF;

-- Step 3: Receiver এ টাকা যোগ করো
UPDATE accounts SET balance = balance + 5000
WHERE account_id = 202;

-- Step 4: Audit log রাখো
INSERT INTO transfer_log (from_id, to_id, amount, created_at)
VALUES (101, 202, 5000, NOW());

COMMIT;  -- সব ঠিক থাকলে permanent save — Durability!
Indexing

Indexing — Database এর Superpower

Index হলো book এর index page এর মতো। পুরো book না পড়ে সরাসরি page এ যাওয়া যায়। Database এ index না থাকলে প্রতিটি query তে পুরো table scan করতে হয় — ১০ লক্ষ row এ এটা অনেক slow।

✅ Index দাও যখন

WHERE clause এ frequently query হয়। High cardinality column (email, phone)। Foreign key columns। Read-heavy workload।

❌ Index এড়াও যখন

Write-heavy table (INSERT/UPDATE slow হয়)। ১০০০ row এর কম small table। Low cardinality (true/false, status)। Rarely queried columns।

SQLindexing-examples.sql
-- 1. Basic index — single column
CREATE INDEX idx_users_email ON users(email);

-- 2. Composite index — দুটো column একসাথে filter করলে
--    WHERE email = ? AND created_at > ?
CREATE INDEX idx_users_email_date ON users(email, created_at);

-- 3. Partial index — শুধু active users (ছোট এবং fast)
CREATE INDEX idx_active_users ON users(email) WHERE status = 'active';

-- 4. Query plan দেখো — index use হচ্ছে কি?
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'test@example.com';
-- "Index Scan" → ভালো ✅  |  "Seq Scan" → index নেই ⚠️

-- N+1 Problem fix — JOIN দিয়ে single query
-- ❌ BAD: 100 users এর জন্য 100টা আলাদা query
-- ✅ GOOD: একটা JOIN query
SELECT u.name, COUNT(p.id) as post_count
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
GROUP BY u.id;
Database Scaling

Sharding এবং Replication

Read Replica — Read Load কমাও

সব database operation এর ৮০-৯০% হলো READ। Primary database এর copy (Replica) রাখো — Read requests Replica তে পাঠাও। Primary শুধু Write handle করবে। এভাবে Primary এর load অনেক কমে।

Sharding — Write Load ভাগ করো

একটা বড় table কে multiple databases এ ভাগ করা। User ID 1-1M → DB1, 1M-2M → DB2। প্রতিটি shard আলাদাভাবে serve করতে পারে।

Approachকীভাবে কাজ করেকখন ব্যবহার করো
Read ReplicaPrimary → Replica async sync। Read → ReplicaRead-heavy app (80%+ read)
Vertical Shardingআলাদা table → আলাদা DB (users DB, orders DB)Microservices separation
Horizontal Shardingএকই table rows ভাগ করে multiple DB তেBillions of rows, write scale
⚠️ Sharding এর সমস্যা
Cross-shard JOIN কঠিন — User A Shard 1 এ, User B Shard 3 এ, এদের data JOIN করতে application layer এ করতে হয়। Resharding complex। Hotspot shard হতে পারে। তাই Sharding সবচেয়ে শেষে করো — আগে Read Replica, Caching, Query optimization দিয়ে দেখো।
Node.jsdb-pool-cache.js
const { Pool } = require('pg');
const redis = require('redis');

// Connection Pool — connection বারবার create করা expensive!
const primaryDB = new Pool({ host: 'primary-db', max: 20 });
const replicaDB = new Pool({ host: 'replica-db', max: 40 });
const cache = redis.createClient();

// Cache-aside Pattern — সবচেয়ে common caching strategy
async function getUser(userId) {
  const cacheKey = `user:${userId}`;

  // 1. Cache check করো
  const cached = await cache.get(cacheKey);
  if (cached) return JSON.parse(cached);  // Cache HIT ✅

  // 2. Cache miss — Replica থেকে read (Primary এ load দিও না)
  const result = await replicaDB.query(
    'SELECT * FROM users WHERE id = $1', [userId]
  );
  const user = result.rows[0];

  // 3. Cache এ store করো (TTL: 1 hour)
  await cache.setEx(cacheKey, 3600, JSON.stringify(user));

  return user;
}

// Write — Primary তে
async function updateUser(userId, data) {
  await primaryDB.query('UPDATE users SET name=$1 WHERE id=$2', [data.name, userId]);
  await cache.del(`user:${userId}`);  // Cache invalidate করো!
}
Real World Examples

বড় কোম্পানিগুলো কীভাবে করেছে

📱 Instagram এর Database Journey

Instagram শুরু করেছিল PostgreSQL দিয়ে। User base ১০M থেকে ১B+ হওয়ার সময় PostgreSQL শarding করে। User ID based consistent hashing দিয়ে ১০০০+ shards। Photos (media files) আলাদাভাবে S3 (object storage) তে রাখে — database তে শুধু metadata। Django ORM এর উপরে custom sharding layer।

🏦 bKash / Banking — ACID Critical

Financial systems এ SQL database অপরিহার্য। Transaction isolation level "SERIALIZABLE" রাখতে হয়। Double-spending prevent করতে database-level locks। Audit log সব transactions এর। Strong consistency mandatory — কোনো eventual consistency চলবে না।

💡 Polyglot Persistence
Real production systems এ এক ধরনের database দিয়ে সব হয় না। Instagram: PostgreSQL (users/follow) + Cassandra (activity feed) + Redis (session/cache) + S3 (photos)। প্রতিটির আলাদা strength কে কাজে লাগানো হয়। এটাই Polyglot Persistence।

B Data integrity maintenance
C Concurrent transactions একে অন্যকে affect করে না

সঠিক উত্তর: C — Isolation মানে প্রতিটি transaction মনে করে সে একা কাজ করছে। দুটো transaction একই account এ simultaneously কাজ করলেও একটা অন্যটার dirty data দেখবে না। Race condition prevent করে।
Question 02 / 10
Database Index এর internal data structure সাধারণত কোনটি?
A Linked List
B B-Tree (Balanced Tree)
C Stack
D Array
সঠিক উত্তর: B — বেশিরভাগ databases B-Tree বা B+Tree use করে। Balanced tree তে O(log n) search, insert, delete। ১ million rows এ মাত্র ~20 comparisons লাগে। Leaf nodes এ actual data pointer থাকে।
Question 03 / 10
Read Replica এর primary use কি?
A Read queries distribute করে Primary এর load কমানো
B Write performance বাড়ানো
C Data sharding করা
D Schema migration করা
সঠিক উত্তর: A — ৮০-৯০% database operations হলো READ। Read replicas দিয়ে এই load distribute করা যায়। Write Primary তে যায়, async এ Replicas তে sync হয়। Primary এর CPU/IO load অনেক কমে।
Question 04 / 10
N+1 Query Problem কি এবং কীভাবে fix করবে?
A Database crash হওয়া — connection pool বাড়িয়ে fix করো
B Schema design এর সমস্যা — Sharding করো
C Index না থাকা — Index যোগ করো
D 100 user এর জন্য 100টা আলাদা query — JOIN বা eager loading দিয়ে fix করো
সঠিক উত্তর: D — N+1: প্রথমে N users fetch করো, তারপর প্রতিটির জন্য আলাদা query। ১০০ users মানে ১০১ queries! Fix: একটাই JOIN query ব্যবহার করো, অথবা ORM এ eager loading (include/populate)।
Question 05 / 10
Redis কোন type এর database?
A Relational database
B Document database
C In-memory Key-Value store
D Graph database
সঠিক উত্তর: C — Redis = Remote Dictionary Server। Memory তে data store করে বলে microseconds এ respond করে। Caching, Sessions, Leaderboards, Pub/Sub এর জন্য perfect। Disk এ backup রাখতে পারে (RDB/AOF) but primarily in-memory।
Question 06 / 10
Connection Pool কেন দরকার?
A Database encrypt করতে
B নতুন DB connection create করা expensive — pool এ reuse করা হয়
C Multiple databases শুধুমাত্র একসাথে query করতে
D Automatic sharding করতে
সঠিক উত্তর: B — DB connection create করা expensive (~50-100ms, TCP handshake + auth)। Pool এ pre-created connections থাকে। Request আসলে pool থেকে নেয়, শেষে return করে। Max pool size = CPU cores × 2-4 হলে good practice।
Question 07 / 10
SQL Injection prevent করার সবচেয়ে effective way কোনটি?
A HTTPS ব্যবহার করা
B Password hashing করা
C Input এর length validate করা
D Parameterized queries / Prepared statements ব্যবহার করা
সঠিক উত্তর: D — Parameterized queries তে SQL code এবং data আলাদা থাকে। User input কখনো SQL code হিসেবে treat হয় না। ORM ব্যবহার করলেও automatically parameterized হয়। String concatenation দিয়ে query build করা সবচেয়ে dangerous।
Question 08 / 10
Denormalization কখন করা উচিত?
A Read-heavy workload এ JOIN কমিয়ে query performance improve করতে
B সবসময়, কারণ data সহজে পাওয়া যায়
C Write-heavy workload এ
D Storage বাঁচাতে
সঠিক উত্তর: A — Normalization = কম redundancy, data integrity। Denormalization = JOIN কমানো, read speed বাড়ানো (redundancy ইচ্ছাকৃতভাবে রাখা)। Analytics, reporting, read-heavy apps এ beneficial। Write-heavy এ unnecessary কারণ update করতে বেশি জায়গায় change করতে হয়।
Question 09 / 10
Sharding এর সবচেয়ে বড় challenge কোনটি?
A Read performance কমে যায়
B Index কাজ করে না
C Cross-shard JOIN কঠিন এবং expensive
D Backup নেওয়া যায় না
সঠিক উত্তর: C — Sharding এর সবচেয়ে বড় challenge হলো cross-shard operations। User A Shard 1 এ, User B Shard 3 এ — তাদের data JOIN করতে application layer এ manually করতে হয়, যেটা complex এবং slow।
Question 10 / 10
Cache-aside pattern কীভাবে কাজ করে?
A সব data সবসময় cache এ রাখো, DB use করো না
B প্রথমে cache check করো, miss হলে DB থেকে নিয়ে cache এ store করো
C Write করার সাথে সাথে cache update করো
D Cache এবং DB কে synchronously update করো
সঠিক উত্তর: B — Cache-aside (Lazy loading): Read → Cache check → Hit? Return। Miss? → DB query → Cache store → Return। Write → DB update → Cache delete (invalidate)। সবচেয়ে common pattern। Simple এবং effective।
