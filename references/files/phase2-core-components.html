Topic 01

⚡ Caching Strategy

Cache হলো system design এর সবচেয়ে powerful tool। সঠিকভাবে ব্যবহার করলে system ১০-১০০ গুণ ফাস্ট হয়। একটা database query ১০০ms নিলে সেটা cache থেকে আসলে নেয় মাত্র ১ms।

📌 Definition
Cache হলো একটা fast storage layer যেটা frequently accessed data কে temporarily রাখে যাতে slow storage (database, disk) থেকে বারবার আনতে না হয়।

Cache কোথায় কোথায় থাকে?

Cache Patterns — ৪টা সবচেয়ে গুরুত্বপূর্ণ

1. Cache-Aside (Lazy Loading) — সবচেয়ে বেশি ব্যবহৃত

Application নিজেই cache manage করে। Cache miss হলে DB থেকে আনে এবং cache এ রাখে।

Pythoncache_aside.py
import redis, json
r = redis.Redis(host='localhost', port=6379)

def get_user(user_id):
    # Step 1: Cache check করো
    cached = r.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)  # Cache HIT ✅
    
    # Step 2: Cache miss — DB থেকে আনো
    user = db.query("SELECT * FROM users WHERE id=?", user_id)
    
    # Step 3: Cache এ রাখো (TTL = 1 ঘণ্টা)
    r.setex(f"user:{user_id}", 3600, json.dumps(user))
    
    return user  # Cache MISS, DB থেকে আনলাম

def update_user(user_id, data):
    db.update("UPDATE users SET ...", data)
    r.delete(f"user:{user_id}")  # Cache invalidate করো!

2. Write-Through — Data consistency বেশি

Write হলেই একসাথে cache এবং DB update। Data loss নেই কিন্তু write latency বেশি।

Pythonwrite_through.py
def save_user(user_id, data):
    # Cache এবং DB একসাথে update
    r.set(f"user:{user_id}", json.dumps(data))
    db.save(user_id, data)
    # ✅ Always consistent কিন্তু write টু বার হয়

3. Write-Back (Write-Behind) — Fast write, risky

Cache এ লেখে, DB তে async এ লেখে পরে। অনেক fast কিন্তু crash হলে data হারাতে পারে।

4. Read-Through — Cache layer সব handle করে

Application শুধু cache কে জিজ্ঞেস করে। Cache নিজেই DB থেকে আনে। Application কে logic লিখতে হয় না।

Cache Eviction Policies

Policyকীভাবে কাজ করেকখন ব্যবহার করো
LRUসবচেয়ে কম recently used item বাদ দাওসাধারণ use case, সবচেয়ে popular
LFUসবচেয়ে কম frequently used item বাদ দাওHot data রাখতে চাইলে
TTLনির্দিষ্ট সময় পর expire হয়Time-sensitive data (session, OTP)
FIFOযে আগে ঢুকেছে সে আগে বের হয়Simple queue-like scenarios

Cache Problems — এগুলো না জানলে interview এ ফেল

🌊 Cache Stampede

Popular cache key expire হলে হাজার হাজার request একসাথে DB তে যায়। Fix: Mutex lock বা probabilistic early expiration।

💀 Cache Penetration

Non-existent data বারবার request হলে সবসময় DB hit হয়। Fix: Bloom filter বা null value cache করো।

🌪️ Cache Avalanche

অনেক cache key একসাথে expire হলে DB overwhelmed। Fix: Random TTL jitter যোগ করো।

💾 Cache Inconsistency

Cache এবং DB এর data আলাদা হয়ে যায়। Fix: Cache invalidation strategy ঠিক করো।

    ttl = base_ttl + jitter

Rate Limiting — গভীরে জানো

Python + Redisrate_limiter.py
import time

def is_rate_limited(user_id, limit=100, window=60):
    """Sliding Window Counter — সবচেয়ে accurate"""
    now = time.time()
    key = f"rate:{user_id}"
    
    pipe = r.pipeline()
    # পুরনো requests remove করো
    pipe.zremrangebyscore(key, 0, now - window)
    # Current request add করো
    pipe.zadd(key, {str(now): now})
    # Count করো
    pipe.zcard(key)
    pipe.expire(key, window)
    results = pipe.execute()
    
    request_count = results[2]
    return request_count > limit  # True = blocked

💾 Database Sharding & Replication

Replication — Data Redundancy

Replication মানে same data কে multiple servers এ রাখা। Availability বাড়ে, read performance বাড়ে।

TypeকীভাবেProsCons
Master-SlaveWrite→Master, Read→SlaveRead scale, backupReplication lag, single write point
Master-Masterদুটোতেই writeHigh availabilityConflict resolution complex
SynchronousAll replicas confirm করলেই successNo data lossHigh latency
AsynchronousMaster ok হলেই successLow latencyPotential data loss

Database Sharding — কেন দরকার?

Replication read scale করে। কিন্তু write scale করতে হলে দরকার Sharding। Database কে horizontal partitions (shards) এ ভাগ করো।

Sharding Strategies

📊 Range-Based

user_id 1-1M → Shard 1, 1M-2M → Shard 2। Simple কিন্তু hotspot possible।

🔢 Hash-Based

hash(user_id) % N → shard number। Even distribution কিন্তু range query কঠিন।

🗺️ Geo-Based

Bangladesh users → Shard-BD, India → Shard-IN। Latency কমায়।

📁 Directory-Based

Lookup table দিয়ে কোন entity কোন shard এ আছে। Flexible কিন্তু lookup overhead।

Consistent Hashing — Sharding এর Best Solution

📌 Why Consistent Hashing?
Normal hashing: N server থেকে N+1 server এ গেলে সব key রemap হয় — cache miss storm।

Consistent hashing: N+1 server যোগ করলে শুধু একটা server এর keys shift হয়। Redis Cluster, Cassandra, DynamoDB এটা ব্যবহার করে।
Pythonconsistent_hashing.py
import hashlib, bisect

class ConsistentHashRing:
    def __init__(self, replicas=100):
        self.replicas = replicas  # Virtual nodes
        self.ring = {}
        self.sorted_keys = []
    
    def add_server(self, server):
        for i in range(self.replicas):
            key = self._hash(f"{server}:{i}")
            self.ring[key] = server
            bisect.insort(self.sorted_keys, key)
    
    def get_server(self, data_key):
        h = self._hash(data_key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]
    
    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)
⚠️ Sharding Challenges
Cross-shard queries: JOIN করা কঠিন — সব user এর friends যদি আলাদা shard এ থাকে
Hotspot: Celebrity user এর shard overloaded — Virtual nodes দিয়ে fix
Re-sharding: Data grow করলে নতুন shard এ migrate করতে হয় — Consistent hashing দিয়ে minimize করো

D FIFO
সঠিক: A — WebSocket connection অনেকক্ষণ open থাকে। Round Robin তে একটা server এ ১০০টা WebSocket, অন্যটায় ১০টা short HTTP — unfair। Least Connections সবসময় কম busy server এ পাঠায়।
প্রশ্ন 06 — Message Queue
Kafka আর RabbitMQ এর সবচেয়ে বড় পার্থক্য কোনটি?
A Kafka faster, RabbitMQ slower
B RabbitMQ open source, Kafka না
C Kafka message retain করে replay possible, RabbitMQ consume হলে delete
D Kafka শুধু cloud এ চলে
সঠিক: C — Kafka হলো event log/stream। Messages days বা weeks রাখতে পারো, replay করতে পারো, নতুন consumer পুরনো events পেতে পারে। RabbitMQ হলো traditional queue — consume হলে gone।
প্রশ্ন 07 — Message Queue
Dead Letter Queue (DLQ) এর উদ্দেশ্য কী?
A Deleted messages backup রাখা
B Max retry পরেও process না হওয়া messages আলাদা করা
C Old messages archive করা
D Priority messages আলাদা queue এ রাখা
সঠিক: B — DLQ = Poison message handler। কোনো message বারবার fail হলে সেটা main queue থেকে সরিয়ে DLQ তে পাঠাও। এতে main queue block হয় না। Team পরে manually investigate করতে পারে।
প্রশ্ন 08 — API Design
User এর orders দেখার জন্য REST API endpoint কেমন হওয়া উচিত?
A GET /getUserOrders?userId=123
B POST /orders/getByUser/123
C GET /orderList/123
D GET /users/123/orders
সঠিক: D — REST এ nested resources parent/id/child format এ। "User 123 এর orders" = /users/123/orders। Verb in URL (getUserOrders) করা যাবে না। GET with POST body ও না। Noun-based, hierarchical।
প্রশ্ন 09 — API Design
gRPC কোন scenario তে REST এর চেয়ে ভালো?
A Internal microservices communication যেখানে low latency দরকার
B Public API যা browser থেকে access হবে
C Simple CRUD operations
D Mobile apps
সঠিক: A — gRPC: HTTP/2 + Protobuf binary = অনেক fast, type-safe, streaming support। Internal service-to-service এর জন্য ideal। Browser থেকে সরাসরি gRPC কঠিন (grpc-web দরকার হয়)।
প্রশ্ন 10 — DB Sharding
Consistent Hashing কেন regular modulo hashing এর চেয়ে ভালো?
A Consistent Hashing faster
B Consistent Hashing কম memory নেয়
C Server add/remove হলে minimum data remapping হয়
D Consistent Hashing এ collision নেই
সঠিক: C — Normal hash: 4→5 server হলে সব key remap। Consistent hashing: শুধু ~1/N keys move করে। Cache cluster তে এটা critical — নইলে সব cache miss হয়ে DB ডুবে যাবে।
প্রশ্ন 11 — DB Replication
Master-Slave replication এ কোন সমস্যাটা সবসময় থাকে?
A Slave write করতে পারে না
B Replication lag — Slave এ data সামান্য পুরনো থাকতে পারে
C Master down হলে data হারিয়ে যায়
D Slave থেকে read করলে error হয়
সঠিক: B — Async replication এ Master এ write হলে Slave এ পৌঁছাতে কিছু milliseconds লাগে। এই সময় Slave পড়লে stale data পাবে। Critical reads (just-wrote data) সবসময় Master থেকে পড়ো।
প্রশ্ন 12 — CDN
CDN কোন ধরনের content এর জন্য সবচেয়ে উপকারী?
A Dynamic user-specific data
B Database queries
C Real-time chat messages
D Static assets: images, CSS, JS, videos
সঠিক: D — CDN static content cache করে globally distribute করে। Dynamic/personalized content CDN এ cache করা যায় না (প্রতি user এর জন্য আলাদা)। Netflix, YouTube এর video files CDN থেকে serve হয়।
প্রশ্ন 13 — CDN & Storage
Pre-signed URL এর সুবিধা কী?
A User সরাসরি S3 তে upload করতে পারে, backend server এর bandwidth বাঁচে
B URL permanent হয়ে যায়
C S3 faster হয়
D File automatically compress হয়
সঠিক: A — Pre-signed URL দিয়ে user browser থেকে সরাসরি S3 তে upload করে। 100MB file আগে server এ আসতো, এখন S3 তে সরাসরি। Server এর RAM, bandwidth, processing সব বাঁচে।
প্রশ্ন 14 — Scenario
একটা news website এ breaking news দেখানো হচ্ছে — ১০ লক্ষ user একসাথে same article পড়ছে। সবচেয়ে effective optimization কোনটা?
A Database index add করো
B ১০টা web server add করো
C Article Redis cache করো এবং CDN এ push করো
D Database server বড় করো
সঠিক: C — Same article ১০ লক্ষ বার DB থেকে আনা = DB ডুবে যাওয়া। Redis cache এ একবার রাখো, সবাই সেখান থেকে পায়। CDN এ push করলে globally fast serve হবে। Web server বাড়ালেও DB bottleneck থাকবে।
প্রশ্ন 15 — Architecture
E-commerce order service এ payment processing 3 seconds নেয়। User কে 3 second wait করানো UX খারাপ। কী করবে?
A Payment service এর server বড় করো
B Message queue দিয়ে async করো — "Order received" দেখাও, payment background এ হোক
C Payment skip করো
D User কে 3 second loading দেখাও
সঠিক: B — Classic async pattern। Order place হলে queue তে message পাঠাও, user কে "Order received!" দেখাও। Payment service queue থেকে নিয়ে process করে। Success হলে notification পাঠাও। Amazon, Flipkart এভাবেই করে।
