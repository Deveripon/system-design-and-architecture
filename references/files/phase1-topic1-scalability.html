Scalability
Fundamentals

⏱ ৪৫-৬০ মিনিট 📊 Junior Level 🏗️ Architecture
Why It Matters

Scalability কেন শিখতে হবে?

তুমি একটা app বানালে। শুরুতে ১০০ user। সব ঠিকঠাক চলছে। কিন্তু হঠাৎ TechCrunch এ feature হলো — ১০ লক্ষ user একসাথে ঢুকলো। Server crash। App down।

এই সমস্যার নাম হলো Scalability problem। একজন system engineer হিসেবে তোমাকে আগে থেকেই plan করতে হবে — আমার system যদি ১০x, ১০০x, ১০০০x বড় হয়, তখনও কি এটা কাজ করবে?

📌 Definition
Scalability হলো একটা system এর ক্ষমতা — বাড়তি load (users, data, requests) handle করার জন্য নিজেকে বড় করার ক্ষমতা, এবং performance ঠিক রাখার ক্ষমতা।
Vertical Scaling

Scale Up — বড় মেশিন কিনো

Vertical scaling মানে তোমার existing server কে আরো powerful বানানো। বেশি RAM, বেশি CPU, বেশি storage।

কখন Vertical Scaling সঠিক?

✅ ভালো situation

Database server — PostgreSQL, MySQL। Single-threaded workload। Code change করা সম্ভব না। Quick fix দরকার।

❌ খারাপ situation

Already সবচেয়ে বড় machine কিনে ফেলেছো। Hardware limit আছে। Cost অনেক বেশি। Single point of failure।

⚠️ Vertical Scaling এর সীমাবদ্ধতা
পৃথিবীর সবচেয়ে বড় single server এও limit আছে। AWS এর biggest instance (u-24tb1.metal) এর ৪৪৮ CPU, ২৪TB RAM — এর বেশি আর যাওয়া সম্ভব না। আর এই machine ঘণ্টায় $200+ এর বেশি।
Horizontal Scaling

Scale Out — বেশি মেশিন যোগ করো

Horizontal scaling মানে একটা বড় মেশিনের বদলে অনেকগুলো ছোট মেশিন ব্যবহার করা। Load টা সবার মধ্যে ভাগ হয়ে যায়।

💡 Key Insight
Horizontal scaling এর সবচেয়ে বড় সুবিধা হলো theoretically unlimited — তুমি যতখুশি server যোগ করতে পারো। Google, Facebook এর লক্ষ লক্ষ server এভাবেই কাজ করে।

Stateless vs Stateful — গুরুত্বপূর্ণ পার্থক্য

Horizontal scaling সফলভাবে করতে হলে তোমার servers কে stateless হতে হবে। মানে, যেকোনো server যেকোনো request handle করতে পারবে।

বিষয় Stateless ✅ Stateful ❌
Session data Redis/DB তে রাখো Server memory তে রাখো
Scale করা যায়? সহজেই যায় কঠিন, sticky session লাগে
Server crash হলে? অন্য server handle করে User এর session হারিয়ে যায়
উদাহরণ REST API, JWT auth Server-side sessions
Node.js stateless-auth.js 
// ❌ STATEFUL — এটা করো না (horizontal scaling এ সমস্যা)
const sessions = {}; // Server memory তে session রাখা = BAD

app.post('/login', (req, res) => {
  sessions[userId] = { loggedIn: true, cart: [...] };
  // User 2nd request করলে অন্য server এ গেলে session নেই!
});

// ✅ STATELESS — এটা করো (horizontal scaling এর জন্য ready)
const redis = require('redis');
const jwt = require('jsonwebtoken');

app.post('/login', async (req, res) => {
  // Session Redis এ রাখো — যেকোনো server access করতে পারবে
  await redis.set(`session:${userId}`, JSON.stringify(sessionData));
  
  // অথবা JWT use করো — server এ কিছুই রাখতে হবে না
  const token = jwt.sign({ userId, role }, process.env.SECRET);
  res.json({ token });
});
Deep Comparison

Vertical vs Horizontal — সম্পূর্ণ তুলনা

বিষয় Vertical Scaling (Scale Up) Horizontal Scaling (Scale Out)
সংজ্ঞা একটা মেশিন বড় করো বেশি মেশিন যোগ করো
Cost Exponentially বাড়ে Linearly বাড়ে
Limit আছে? হ্যাঁ, hardware limit না, theoretically unlimited
Downtime লাগে? হ্যাঁ, upgrade করতে না, live এ add করা যায়
Single point of failure হ্যাঁ (risky) না (fault tolerant)
Code change লাগে? না হ্যাঁ (stateless করতে হয়)
কখন ব্যবহার করো DB servers, quick fix Web servers, API, large scale
🎯 Interview এ এটা বলো
"আমি সবসময় vertical scaling দিয়ে শুরু করবো কারণ এটা simpler। কিন্তু যখন hardware limit আসবে বা single point of failure acceptable না হবে, তখন horizontal scaling এ shift করবো।" — এটাই সঠিক industry approach।
Real World Examples

বড় কোম্পানিগুলো কীভাবে করেছে

🐦 Twitter এর গল্প

২০০৮ সালে Twitter Rails monolith ছিল — একটাই server। ২০০৯ এর Oscars broadcast এ tweet storm এলো, সব crash। তারপর তারা horizontal scaling + stateless architecture তে গেলো। এখন হাজার হাজার server চলে।

📈 Stack Overflow এর ব্যতিক্রম

Stack Overflow শুধু Vertical Scaling ব্যবহার করে এবং মাত্র কয়েকটা server দিয়ে ১ কোটি+ daily user handle করে। কারণ তাদের কাজ read-heavy (বেশিরভাগ লোক শুধু পড়ে, লেখে কম) এবং heavy caching ব্যবহার করে।

💡 এটাই মূল শিক্ষা
"Always horizontal scale" — এটা ভুল। Use case বুঝে decide করো। Stack Overflow মাত্র ৯টা web server দিয়ে বিশাল traffic handle করে। কিন্তু Netflix এর ১০,০০০+ server দরকার।

Load Balancer কীভাবে কাজ করে (Quick Look)

Horizontal scaling কাজ করে Load Balancer এর মাধ্যমে। Load Balancer incoming traffic নেয় এবং সব server এর মধ্যে ভাগ করে দেয়।

Nginx Config nginx.conf 
# Simple Nginx Load Balancer config
upstream backend {
    # Round Robin (default) — সব server এ সমান ট্র্যাফিক
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;
    server 192.168.1.12:3000;
    
    # Health check — কোনো server down হলে automatically বাদ দাও
    keepalive 32;
}

server {
    listen 80;
    
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
Advanced Patterns

Scaling Patterns যা জানতে হবে

1. Database Scaling Problem

Web servers horizontal scale করা সহজ। কিন্তু database? সব server তো একই DB তে লিখছে!

Approach কীভাবে কাজ করে কখন ব্যবহার করো
Read Replica Write → Master, Read → Replica Read-heavy apps (blogs, news)
Database Sharding Data টুকরো টুকরো করে ভাগ করো Massive data (billions of rows)
Caching Layer Redis/Memcached দিয়ে DB hit কমাও সবসময় (DB এর সামনে cache রাখো)

2. The Bottleneck Rule

📌 Core Principle
System সবসময় তার সবচেয়ে slow component এর speed এ চলে। তুমি ১০০০ web server দিলেও যদি database slow থাকে, পুরো system slow। Bottleneck খোঁজো, সেটা fix করো।

3. Auto-Scaling

Modern cloud এ (AWS, GCP) তুমি auto-scaling set করতে পারো। Traffic বাড়লে automatically নতুন server আসবে, কমলে চলে যাবে। এতে cost optimize হয়।

AWS CLI auto-scaling.sh 
# AWS Auto Scaling Group তৈরি করা
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name "my-app-asg" \
  --min-size 2 \        # সবসময় minimum 2 server
  --max-size 20 \       # maximum 20 server পর্যন্ত বাড়াও
  --desired-capacity 3  # শুরুতে 3 server

# Scaling Policy: CPU > 70% হলে নতুন server যোগ করো
aws autoscaling put-scaling-policy \
  --policy-name "scale-out" \
  --scaling-adjustment 2 \  # 2টা server যোগ করো
  --adjustment-type ChangeInCapacity

Knowledge Check — MCQ

Score: 0/10
প্রশ্ন 01
একটি e-commerce app এর database server এর CPU usage ৯০% হয়ে গেছে। তুমি দ্রুততম সমাধান হিসেবে কী করবে?
A আরও ৫টা application server যোগ করবো
B Database server এ বেশি CPU/RAM add করবো (Vertical Scale)
C নতুন database software কিনবো
D App এর code পুরো rewrite করবো
সঠিক উত্তর: B — Database এর CPU bottleneck এর জন্য দ্রুততম সমাধান হলো vertical scaling (বড় মেশিন)। Application servers বাড়ালে database এর load আরও বাড়বে। Database horizontal scaling (sharding) complex এবং সময় লাগে।
প্রশ্ন 02
Horizontal scaling সফলভাবে কাজ করার জন্য সবচেয়ে গুরুত্বপূর্ণ কোনটি?
A সব server একই hardware হওয়া
B Load balancer ব্যবহার করা
C Application কে Stateless বানানো
D Database server বড় করা
সঠিক উত্তর: C — Stateless architecture ছাড়া horizontal scaling কাজ করবে না। Load balancer দরকার (B) কিন্তু সেটা sufficient না — stateful app এ sticky session এর সমস্যা হবে। Stateless মানে যেকোনো server যেকোনো request handle করতে পারে।
প্রশ্ন 03
Stack Overflow মাত্র কয়েকটা server দিয়ে কোটি user handle করে। এর প্রধান কারণ কী?
A তারা খুব সস্তা server ব্যবহার করে
B Read-heavy workload এবং heavy caching ব্যবহার
C তাদের user সংখ্যা আসলে কম
D তারা serverless ব্যবহার করে
সঠিক উত্তর: B — Stack Overflow এর ৯৫%+ traffic হলো read (প্রশ্ন পড়া)। Write (প্রশ্ন করা, উত্তর দেওয়া) অনেক কম। Heavy caching দিয়ে same content বারবার DB থেকে না এনে cache থেকে serve করে। তাই কম server দিয়েই চলে।
প্রশ্ন 04
নিচের কোনটি Vertical Scaling এর সবচেয়ে বড় সমস্যা?
A Setup করতে কঠিন
B Load balancer দরকার হয়
C Code change করতে হয়
D Hardware limit আছে এবং Single point of failure
সঠিক উত্তর: D — Vertical scaling এর দুটো critical সমস্যা: (১) পৃথিবীর সবচেয়ে বড় server এরও limit আছে। (২) একটাই server থাকায় সেটা down হলে পুরো app down — এটাই Single Point of Failure (SPOF)।
প্রশ্ন 05
একটি social media app এ user login করলে server memory তে session রাখা হয়। এই app কে horizontal scale করলে কী সমস্যা হবে?
A User যদি ভিন্ন server এ যায় তাহলে logged out হয়ে যাবে
B Server বেশি RAM use করবে
C Database slow হয়ে যাবে
D Load balancer কাজ করবে না
সঠিক উত্তর: A — এটাই stateful architecture এর classic problem। User প্রথম request করে Server 1 এ — session সেখানে আছে। পরের request যায় Server 2 এ — সেখানে session নেই, তাই user কে logged out মনে হয়। Solution: Redis এ session রাখো অথবা JWT ব্যবহার করো।
প্রশ্ন 06
AWS Auto Scaling এর কাজ কী?
A Automatically database backup নেওয়া
B Code automatically deploy করা
C Traffic অনুযায়ী automatically server সংখ্যা বাড়ানো-কমানো
D Server এর hardware upgrade করা
সঠিক উত্তর: C — Auto Scaling group define করে min/max server সংখ্যা। CPU বা request rate বেশি হলে automatically নতুন instance চালু হয়, কমে গেলে বন্ধ হয়। এতে cost optimize হয় এবং traffic spike handle হয়।
প্রশ্ন 07
একটা system এর web server গুলো fast কিন্তু database queries ৫ সেকেন্ড নিচ্ছে। আরও web server যোগ করলে কী হবে?
A System অনেক fast হয়ে যাবে
B Database এর উপর চাপ আরও বাড়বে, system আরও slow হবে
C Database automatically fast হয়ে যাবে
D কিছুই পরিবর্তন হবে না
সঠিক উত্তর: B — এটাই Bottleneck Rule এর উদাহরণ। Database যদি bottleneck হয়, তাহলে বেশি web server মানে বেশি DB requests — database আরও overwhelmed হবে। আগে bottleneck (database) fix করো — caching, read replica, বা query optimization।
প্রশ্ন 08
JWT (JSON Web Token) ব্যবহার করলে horizontal scaling সহজ হয় কারণ —
A JWT encryption খুব fast
B JWT database এ রাখা হয়
C JWT শুধু একটা server এ কাজ করে
D JWT stateless — server কে কিছু store করতে হয় না, যেকোনো server verify করতে পারে
সঠিক উত্তর: D — JWT এর সৌন্দর্য হলো self-contained। Token এর মধ্যেই user info আছে। যেকোনো server shared secret দিয়ে token verify করতে পারে — database বা shared memory দরকার নেই। এটাই stateless auth।
প্রশ্ন 09
Nginx load balancer এ Round Robin algorithm কীভাবে কাজ করে?
A Requests একে একে সব server এ পাঠায় — 1→2→3→1→2→3...
B সবচেয়ে কম busy server এ পাঠায়
C User এর IP অনুযায়ী সবসময় একই server এ পাঠায়
D Random server এ পাঠায়
সঠিক উত্তর: A — Round Robin সবচেয়ে simple algorithm। Circular order এ একে একে প্রতিটা server এ request দেয়। সব server সমান powerful হলে এটাই best। "Least Connections" algorithm busy server avoid করে — সেটা B।
প্রশ্ন 10
একটি banking app design করছো — ৫০০ concurrent users এবং ভবিষ্যতে ৫০,০০০ users হতে পারে। কোন approach নেবে?
A এখনই ১০০টা server নিয়ে শুরু করো
B Vertical scaling করতে থাকো, horizontal এর কথা পরে ভাববো
C Stateless architecture দিয়ে শুরু করো, ছোট করে শুরু করো কিন্তু horizontal scale করার জন্য ready রাখো
D NoSQL database ব্যবহার করলেই scaling সমস্যা সমাধান হবে
সঠিক উত্তর: C — এটাই pragmatic approach। এখনই ১০০ server waste। শুধু vertical scaling এ আটকে থাকলে পরে migrate করতে কষ্ট হবে। সঠিক পথ: ছোট শুরু + stateless architecture + load balancer ready রাখো। এটাই "scale when needed" philosophy।

Hands-On Project Task

এটা practical task। Docker থাকলে এখনই করতে পারো। না থাকলে Docker install করে তারপর করো।

Mini Load Balancer Setup

Docker Compose দিয়ে একটা Nginx load balancer এবং ৩টা Node.js server চালাও।

1

Docker install করো

docker.com থেকে Docker Desktop install করো। Windows/Mac/Linux সব সাপোর্ট করে।

2

Simple Node.js server বানাও

server.js — শুধু "Hello from Server X" return করবে, X হবে server এর নাম।

3

docker-compose.yml লেখো

৩টা app service এবং ১টা nginx service define করো।

4

Nginx config লেখো

upstream block এ ৩টা server add করো।

5

Test করো

Browser এ localhost এ বারবার refresh করো — দেখো কোন server respond করছে।

6

Bonus: একটা server kill করো

docker stop দিয়ে একটা server বন্ধ করো। দেখো অন্য দুটো তখনও কাজ করছে কিনা।

docker-compose docker-compose.yml 
version: '3'
services:
  nginx:
    image: nginx:alpine
    ports: ["80:80"]
    volumes: ["./nginx.conf:/etc/nginx/nginx.conf"]
    depends_on: [app1, app2, app3]

  app1:
    build: .
    environment:
      SERVER_NAME: "Server-1"

  app2:
    build: .
    environment:
      SERVER_NAME: "Server-2"

  app3:
    build: .
    environment:
      SERVER_NAME: "Server-3"
🎯 এটা করলে যা শিখবে
Theoretical না — actually দেখবে Load Balancer কীভাবে traffic ভাগ করে। একটা server kill করলে দেখবে system তখনও চলছে। এটাই horizontal scaling এর power।
