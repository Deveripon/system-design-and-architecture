Why This System

Twitter কেন Design শেখা দরকার?

Twitter/X হলো সবচেয়ে complex social media system design interview question গুলোর একটা। এখানে News Feed generation, follower/following graph, real-time trending — এই তিনটা hard problem একসাথে solve করতে হয়।

📌 Core Challenges
Fanout Problem: একজন celebrity যখন tweet করে (Elon Musk: 150M followers), তখন সেই tweet 150M user-এর feed-এ push করতে হবে। এটাই Twitter-এর সবচেয়ে কঠিন scaling problem।

Deep Dive

Fanout Problem — Twitter-এর Hardest Part

যখন কেউ tweet করে, সেই tweet তার সব followers-এর home timeline-এ দেখা যায়। এটা করার দুটো approach আছে।

Approachকীভাবে কাজ করেWrite CostRead CostProblem
Fan-out on Write (Push)Tweet করলেই সব followers-এর cache updateHigh (150M writes)Low (cache hit)Celebrity problem
Fan-out on Read (Pull)Timeline load করলে following-দের tweets pullLowHigh (DB joins)Slow timeline
Hybrid (Twitter's approach)Normal users: push, Celebrity: pull on readBalancedBalancedBest of both
💡 Twitter-এর Real Solution
Twitter threshold ≈ 10,000 followers। এর কম followers হলে fan-out on write (push to all)। বেশি হলে fan-out on read (pull at read time)। এই hybrid approach-ই production-এ use হয়।
Python fanout_service.py 
async def fanout_tweet(tweet_id, author_id, db, cache, queue):
    # Author-এর follower list নাও
    followers = await db.get_followers(author_id)
    
    # Hybrid Fanout Logic
    if len(followers) < 10_000:
        # Normal user: push to all followers' timeline cache
        for follower_id in followers:
            cache.lpush(f"timeline:{follower_id}", tweet_id)
            cache.ltrim(f"timeline:{follower_id}", 0, 799)  # last 800 tweets
    else:
        # Celebrity: push to Kafka, workers will process async
        # At read time, celebrity tweets are fetched separately
        await queue.publish("celebrity-fanout", {
            "tweet_id": tweet_id,
            "author_id": author_id
        })
    
    # Tweet always saved to Tweet DB
    await db.save_tweet(tweet_id, author_id)

📰 Home Timeline — Read Path

Python timeline_service.py 
async def get_home_timeline(user_id, db, cache):
    # Step 1: User-এর pre-built timeline cache থেকে নাও
    cached = cache.lrange(f"timeline:{user_id}", 0, 19)
    
    # Step 2: Celebrity following list check
    celebrities = cache.smembers(f"following_celebrities:{user_id}")
    
    # Step 3: Celebrity tweets আলাদাভাবে fetch করো
    celebrity_tweets = []
    for celeb_id in celebrities:
        tweets = await db.get_recent_tweets(celeb_id, limit=5)
        celebrity_tweets.extend(tweets)
    
    # Step 4: Merge করো, sort by timestamp
    all_tweet_ids = cached + [t.id for t in celebrity_tweets]
    all_tweet_ids.sort(key=lambda x: x.created_at, reverse=True)
    
    return all_tweet_ids[:20]  # Top 20 tweets
Database Choice

Multiple Databases — কোনটা কিসের জন্য?

Twitter-এ single database দিয়ে হয় না। Different use cases-এর জন্য different databases।

Data TypeDatabaseWhy?
Tweets (immutable)CassandraAppend-only, time-series, massive scale
User profilesPostgreSQLACID, relational, profile updates
Follow graphRedis (sorted set) / Neo4jGraph traversal, follower counts
Home timelineRedis (list)Fast reads, pre-computed, in-memory
Search indexElasticsearchFull-text search, hashtag search
Media filesAmazon S3 + CloudFrontObject storage, CDN delivery
AnalyticsHadoop/Spark + HiveBatch processing, trend analysis
⚠️ Tweet Schema Design
Tweets এ created_at দিয়ে partition করো Cassandra-তে। Time-series data — নতুন tweets সবসময় write, পুরনো tweets rarely read। Cassandra-র time-based partitioning perfect fit।
Scaling Decisions

Twitter Scale করার Strategies

Strategy
Timeline Caching: প্রতিটা user-এর last 800 tweets Redis-এ store। 99% reads cache hit → DB আসে না।
Strategy
Async Fanout via Kafka: Tweet POST করলে immediately return। Background-এ Kafka workers fanout করে। User experience fast রাখে।
Strategy
Read Replicas: Tweet DB-তে 1 master + 5 replicas। সব reads replica থেকে → master শুধু writes।
Trade-off
Eventual Consistency: Celebrity tweet হলে সবার timeline-এ 1-2 sec delay হতে পারে। Acceptable trade-off।
Trade-off
Storage Cost: প্রতিটা tweet 800 user-এর cache-এ store মানে 800x storage। Memory expensive কিন্তু latency দিয়ে pay করা worth।

A 1 write (শুধু tweet save)
B 1000 writes

সঠিক উত্তর: C — Fan-out on write মানে প্রতিটা follower-এর timeline cache update করো। 150M followers → 150M cache writes। এটাই "celebrity problem" বা "thundering herd" — তাই Twitter hybrid approach use করে।
Question 02 / 10
Twitter-এর Hybrid Fanout approach-এ celebrity threshold কত followers?
A 1,000 followers
B 10,000 followers
C 1 million followers
D 100,000 followers
সঠিক উত্তর: B — Twitter-এর actual threshold ≈ 10,000 followers। এর কম = fan-out on write (push)। বেশি = fan-out on read (pull at timeline load time)। এই number exact নয়, কিন্তু concept টা interview-এ গুরুত্বপূর্ণ।
Question 03 / 10
Home Timeline-এর জন্য Redis-এ কোন data structure সবচেয়ে appropriate?
A List (LPUSH + LTRIM)
B Hash Map
C Set
D String
সঠিক উত্তর: A — Redis List এ LPUSH দিয়ে নতুন tweet সামনে add করো, LTRIM দিয়ে last 800 রাখো। Ordered by insertion time — timeline-এর জন্য perfect। LRANGE দিয়ে pagination করা যায়।
Question 04 / 10
Tweet data store করার জন্য সবচেয়ে suitable database কোনটা এবং কেন?
A MySQL — ACID transactions দরকার
B MongoDB — flexible schema
C SQLite — simple and fast
D Cassandra — immutable, time-series, massive write scale
সঠিক উত্তর: D — Tweets are immutable (একবার post, rarely edit) এবং time-series (by created_at)। Cassandra partition by time, massive write throughput। Twitter নিজে Cassandra use করে tweet storage-এর জন্য।
Question 05 / 10
Hashtag search (#Python) implement করতে কোন database best?
A Redis
B Elasticsearch
C Cassandra
D PostgreSQL LIKE query
সঠিক উत তর: B — Elasticsearch full-text search-এর জন্য designed। Inverted index দিয়ে hashtag search milliseconds-এ হয়। PostgreSQL LIKE query O(n) — 200M tweets-এ অনেক slow। Twitter Elasticsearch use করে search feature-এ।
Question 06 / 10
Twitter-এ Follow graph (কে কাকে follow করে) store করার জন্য কোনটা ভালো?
A MySQL table
B File system
C Redis Sorted Set / Graph DB
D Array in memory
সঠিক উত্তর: C — Follow graph = social graph। Redis Sorted Set-এ user_id → follower_ids store করো। Follower count, mutual follows efficiently query করা যায়। Large scale-এ Neo4j বা FlockDB (Twitter's own graph DB) use হয়।
Question 07 / 10
Tweet image/video কোথায় store করবে?
A Amazon S3 + CloudFront CDN
B Database BLOB field
C API Server local disk
D Redis cache
সঠিক উত্তর: A — Binary files (images, videos) object storage-এ রাখো। S3 reliable, cheap, scalable। CDN (CloudFront) globally distribute করে — user-এর কাছের edge server থেকে fast delivery। Database-এ BLOB = terrible idea for scale।
Question 08 / 10
Real-time Trending Topics (#Trending) কীভাবে implement করবে?
A প্রতি second DB scan করো
B User vote করবে
C Daily batch job
D Redis Sorted Set + sliding window count
সঠিক উত্তর: D — Redis Sorted Set এ hashtag → last 1 hour count রাখো। নতুন tweet → ZINCRBY করো। ZREVRANGE দিয়ে top trending পাও। Sliding window (last 1 hour) দিয়ে stale trends avoid করো।
Question 09 / 10
Twitter-এর Like count millions হলে race condition prevent করবে কীভাবে?
A Single thread এ সব process করো
B Redis INCR (atomic operation)
C DB transaction lock করো
D Like count approximate রাখো
সঠিক উত্তর: B — Redis INCR atomic operation — concurrent requests-এ race condition নেই। DB-তে SELECT FOR UPDATE lock expensive। Redis INCR lightning fast, atomic, perfect for counters।
Question 10 / 10
Twitter-এ Kafka use করার primary reason কী?
A Primary database হিসেবে
B User authentication এর জন্য
C Async fanout — tweet post এর পরে background-এ distribute করতে
D Image compression এর জন্য
সঠিক উত্তর: C — Tweet POST request immediately return করে। Kafka queue-তে tweet_id push হয়। Background consumers async-ভাবে follower timelines update করে। এই decoupling দিয়ে write latency কমে এবং system resilient হয়।
