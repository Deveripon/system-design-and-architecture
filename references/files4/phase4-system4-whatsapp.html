Why This System

Chat System-এর Unique Challenges

WhatsApp-এ প্রতিদিন 100 billion messages send হয়। এই system-এ সবচেয়ে কঠিন সমস্যা হলো real-time delivery — message পাঠানো সাথে সাথে receiver পাবে, এমনকি receiver offline থাকলেও পরে পাবে।

📌 Core Challenge
Push vs Pull: Traditional HTTP request-response কাজ করে না real-time messaging-এ। Receiver সব সময় "নতুন message আছে?" জিজ্ঞেস করতে পারে না। Server-ই push করতে হবে। এই জন্য WebSocket দরকার।

HTTP request-response model real-time chat-এর জন্য suitable না। WebSocket একটা persistent bidirectional connection রাখে।

    await redis.set(f"user:{user_id}:server", SERVER_ID)
    await redis.set(f"user:{user_id}:online", "true")
    
    try:

            msg = json.loads(raw_msg)

                sender_id=user_id,
                receiver_id=msg['to'],
                content=msg['content']
            )
    finally:
        # Disconnect হলে cleanup
        del connections[user_id]
        await redis.set(f"user:{user_id}:online", "false")
        await redis.set(f"user:{user_id}:last_seen", now())

👥 Group Chat — Multi-receiver Problem

⚠️ Group Chat Fanout
1000 member group-এ 1 message → 999 users-এ deliver করতে হবে। Fan-out problem! Solution: Message DB-তে 1 copy রাখো, per-user read status track করো। Message copy করো না — pointer রাখো।
SQL schema.sql — Group Chat 
-- Group messages: 1 row per message (not per user)
CREATE TABLE group_messages (
  msg_id       UUID PRIMARY KEY,
  group_id     UUID,
  sender_id    UUID,
  content      TEXT,
  sent_at      TIMESTAMP
);

-- Per-user delivery tracking (not message copy)
CREATE TABLE message_receipts (
  msg_id       UUID,
  user_id      UUID,
  delivered_at TIMESTAMP,
  read_at      TIMESTAMP,
  PRIMARY KEY (msg_id, user_id)
);
-- 1000 member group = 1 msg + 999 receipt rows (not 1000 msg copies)

💡 Message Schema in Cassandra
Partition key = (chat_id), Clustering key = (message_id DESC)। এতে latest messages সামনে আসে। Pagination করা সহজ। Time-ordered messages efficiently store হয়।
Scaling Decisions

Chat System Scale করার Strategies

Strategy
Horizontal Chat Servers: লক্ষ লক্ষ WebSocket connections ধরে রাখতে অনেক Chat servers। Load balancer user-কে consistent server-এ route করে (sticky sessions)।
Strategy
Service Discovery via Zookeeper: প্রতিটা Chat server register হয় Zookeeper-এ। User কোন server-এ connected তা Redis-এ cache থাকে।
Strategy
Kafka for Cross-server Routing: User A (Server 1) → User B (Server 5)। Kafka topic per server। Server 5-এর Kafka topic-এ message publish → Server 5 consume করে User B-কে push।
Trade-off
End-to-End Encryption: E2E encryption মানে server message decrypt করতে পারে না। Spam detection, moderation কঠিন হয়। Privacy vs Safety trade-off।

MCQ Test — WhatsApp/Chat System

Score: 0/10
Question 01 / 10
Real-time chat-এর জন্য HTTP polling কেন খারাপ?
A HTTP too slow
B HTTP doesn't support text
C HTTP too expensive
D প্রতিবার নতুন connection, high latency, server-side push নেই
সঠিক উত্তর: D — Polling-এ client বারবার "নতুন message আছে?" জিজ্ঞেস করে। প্রতিবার TCP connection = overhead। Message আসলেও পরের poll পর্যন্ত wait। WebSocket একটা persistent connection রাখে — message আসলেই server push করে।
Question 02 / 10
User A (Chat Server 1) → User B (Chat Server 5) message পাঠাবে কীভাবে?
A Direct HTTP call Server 1 → Server 5
B Kafka/Message Queue দিয়ে route করো
C Database write করো, Server 5 poll করবে
D Impossible — একই server-এ থাকতে হবে
সঠিক উত্তর: B — Kafka topic per server। Server 1 → Kafka topic "server-5" → Server 5 consume করে → User B-কে push। এই pub/sub pattern দিয়ে cross-server messaging হয়। Direct HTTP call করলে tight coupling হয়।
Question 03 / 10
User-এর online/offline status কোথায় store করবে?
A Redis (with TTL)
B MySQL
C Cassandra
D File system
সঠিক উত্তর: A — Redis in-memory, millisecond read। TTL set করো — user disconnect হলে TTL expire হলে automatically "offline"। Heartbeat: every 30 sec Redis key refresh করো। DB-তে write করলে too many writes, too slow।
Question 04 / 10
User offline থাকলে তাকে message deliver করবে কীভাবে?
A Message drop করে দাও
B Sender-কে error দাও
C Message DB-তে store, online হলে fetch করে push
D SMS পাঠাও
সঠিক উত্তর: C — Persistent message store। User comes online → Chat server checks undelivered messages → pushes all pending। এটাই "offline message delivery"। Message কখনো drop হবে না।
Question 05 / 10
1000 member group chat-এ message store কীভাবে করবে?
A 1000 copies — প্রতি user-এর জন্য
B শুধু sender-এর কাছে
C Broadcast করো, store করো না
D 1 copy + per-user delivery receipt table
সঠিক উত্তর: D — 1 message copy in group_messages table। 999 receipt rows in message_receipts table। এতে storage 1000x less। Delivery status per-user track হয়। WhatsApp এই approach use করে।
Question 06 / 10
WhatsApp Messages store করার জন্য কোন database best?
A MySQL
B Cassandra
C Redis
D MongoDB
সঠিক উত্তর: B — Messages = append-only time-series data। Cassandra partition by chat_id, cluster by message_id desc। Massive write throughput। Facebook Messenger এবং WhatsApp দুটোই HBase/Cassandra use করে।
Question 07 / 10
WhatsApp কোন programming language-এ Chat Servers build করেছিল?
A Erlang
B Java
C C++
D Python
সঠিক উত্তর: A — Erlang massive concurrent connections-এর জন্য designed। Lightweight processes (actors), fault-tolerant। WhatsApp 50 engineers দিয়ে 450M users serve করত Erlang-এর কারণে। Telecom industry standard language।
Question 08 / 10
WhatsApp-এর End-to-End Encryption-এর trade-off কী?
A Messages slow হয়
B Messages deliver হয় না
C Server message read করতে পারে না, spam detection কঠিন
D Battery বেশি খরচ হয়
সঠিক উত্তর: C — E2E encryption মানে শুধু sender ও receiver decrypt করতে পারে। Server ciphertext দেখে। এতে user privacy বাড়ে কিন্তু spam, child abuse content detection impossible। Privacy vs Safety — ongoing debate।
Question 09 / 10
Zookeeper Chat System-এ কী কাজে ব্যবহার হয়?
A Message storage
B User authentication
C Video processing
D Service discovery — কোন Chat server available তা track করা
সঠিক উত্তর: D — Zookeeper distributed coordination service। Chat servers register করে। Load balancer জানে কোন servers available। Server down হলে automatically deregister। Kafka-ও internally Zookeeper use করে।
Question 10 / 10
WebSocket connection কতক্ষণ open থাকে?
A শুধু message send/receive-এর সময়
B App open থেকে close পর্যন্ত সবসময়
C প্রতি 1 minute
D শুধু network available হলে
সঠিক উত্তর: B — WhatsApp open থাকলে persistent WebSocket connection। Heartbeat ping/pong প্রতি 30 sec। এই connection দিয়েই messages, delivery receipts, online status update সব push হয়। App close → connection close → Redis "offline"।
