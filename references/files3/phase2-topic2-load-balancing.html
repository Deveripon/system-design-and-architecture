01 — Core Concept

Load Balancer কী?

ধরো একটা ব্যাংকে ১০টা counter আছে কিন্তু সব customer একটা counter এ লাইন দিচ্ছে — বাকি ৯টা খালি। এটাই poor distribution। Load Balancer হলো সেই manager যে বলে দেয় কোন customer কোন counter এ যাবে — সব server এ সমান ভাবে traffic ভাগ করে।

💡 Definition
Load Balancer একটা traffic distribution system যেটা incoming requests কে multiple backend servers এ distribute করে। এতে কোনো single server overload হয় না, system highly available থাকে, এবং horizontal scaling সহজ হয়।
✅ Load Balancer এর সুবিধা
High Availability (একটা server down হলেও অন্যরা চলে), Scalability (সহজে server add করা যায়), Performance (traffic distribute হয়, কোনো server overload হয় না), এবং Zero-downtime deployment (একটা server update করার সময় অন্যরা serve করে)।
02 — L4 vs L7

Layer 4 vs Layer 7 Load Balancing

Load Balancer OSI Model এর দুটো different layer এ কাজ করতে পারে — এটাই L4 এবং L7 এর পার্থক্য।

Layer 4 — Transport Layer (TCP/UDP)

L4 Load Balancer শুধু IP address এবং TCP port দেখে traffic route করে। HTTP packet এর content দেখে না — শুধু network level এ কাজ করে। অনেক fast কিন্তু content-aware routing করতে পারে না।

Layer 7 — Application Layer (HTTP/HTTPS)

L7 Load Balancer HTTP request এর সব content দেখতে পারে — URL path, headers, cookies, query params। এতে intelligent routing করা যায়। Slower than L4 কিন্তু অনেক বেশি powerful।

FeatureLayer 4Layer 7
OSI LayerTransport (TCP/UDP)Application (HTTP)
SpeedVery FastFast (slight overhead)
Content AwarenessNoYes (URL, headers, cookies)
SSL TerminationNoYes
Path-based RoutingNoYes

03 — Algorithms

Load Balancing Algorithms

1. Round Robin

সবচেয়ে simple। Request গুলো একে একে সব server এ পাঠানো হয়: Server 1 → Server 2 → Server 3 → Server 1 → ...

2. Weighted Round Robin

Server এর capacity অনুযায়ী weight দেওয়া হয়। বড় server বেশি request পায়।

3. Least Connections

যে server এ সবচেয়ে কম active connection আছে সেখানে পাঠানো হয়। Long-lived connections (WebSocket, DB) এর জন্য ভালো।

4. IP Hash

Client এর IP address hash করে সবসময় same server এ পাঠানো হয়। Session persistence দরকার হলে (stateful app) এটা কাজ করে।

5. Random

Randomly একটা server choose করে। Simple কিন্তু uneven distribution হতে পারে।

Algorithmকীভাবে কাজ করেBest ForProblem
Round Robinএকে একে সবাইEqual capacity serversDifferent capacity ignore করে
Weighted RRWeight অনুযায়ীMixed capacityStatic weight, dynamic load না
Least Connectionsকম connection এ পাঠাওLong requests, varied loadSlightly complex
IP HashIP → same serverSession persistenceServer down হলে সব session হারায়
RandomRandom choiceSimple, testingHot spots possible
04 — Health Checks

Health Check ও Failover

Load Balancer regularly সব server কে ping করে check করে কোনটা alive। যদি কোনো server respond না করে — সেই server কে automatically বাদ দেওয়া হয়।

⚠️ Health Check Types
TCP Check: শুধু port open আছে কিনা check। HTTP Check: নির্দিষ্ট URL এ GET request করে 200 OK আসে কিনা দেখে। Custom Script: Application-specific logic দিয়ে health check। সাধারণত /health বা /ping endpoint বানানো হয়।
Nginxnginx.conf
upstream backend {
  # Load balancing algorithm
  least_conn;  # or: round-robin (default), ip_hash

  # Servers with weights
  server app1:3000 weight=3;  # more powerful server
  server app2:3000 weight=2;
  server app3:3000 weight=1;

  # Health check (nginx plus feature)
  # keepalive 32;
}

server {
  listen 80;

  location / {
    proxy_pass http://backend;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;

    # Retry on failure
    proxy_next_upstream error timeout;
    proxy_next_upstream_tries 3;
  }
}

06 — Code Examples

Load Balancing Code

Python — Custom Round Robin Implementation

Pythonload_balancer.py
from itertools import cycle
from threading import Lock
import requests

class RoundRobinLoadBalancer:
    def __init__(self, servers: list):
        self.servers = [s for s in servers]
        self.healthy = servers.copy()
        self.pool = cycle(self.healthy)
        self.lock = Lock()

    def get_server(self):
        """Get next healthy server"""
        with self.lock:
            if not self.healthy:
                raise Exception("No healthy servers!")
            return next(self.pool)

    def health_check(self):
        """Check all servers health"""
        newly_healthy = []
        for server in self.servers:
            try:
                resp = requests.get(f"{server}/health", timeout=2)
                if resp.status_code == 200:
                    newly_healthy.append(server)
            except:
                print(f⚠️ {server} unhealthy, removing)

        with self.lock:
            self.healthy = newly_healthy
            self.pool = cycle(self.healthy)

# Usage
lb = RoundRobinLoadBalancer([
    "http://server1:3000",
    "http://server2:3000",
    "http://server3:3000",
])

# Proxy a request

Node.js — Least Connections Algorithm

Node.jsleastConnections.js
class LeastConnectionsLB {
  constructor(servers) {
    // Track connection count per server
    this.servers = servers.map(s => ({
      url: s,
      connections: 0,
      healthy: true
    }));
  }

  getServer() {
    const available = this.servers.filter(s => s.healthy);
    if (!available.length) throw new Error('No servers!');

    // Pick server with minimum active connections
    return available.reduce((min, s) =>
      s.connections < min.connections ? s : min
    );
  }

  async forward(req, res) {
    const server = this.getServer();
    server.connections++;   // Track active

    try {
      const response = await fetch(`${server.url}${req.url}`);
      res.send(await response.text());
    } finally {
      server.connections--;  // Release
    }
  }
}

const lb = new LeastConnectionsLB([
  'http://server1:3001',
  'http://server2:3001',
  'http://server3:3001'
]);

08 — Interview Prep

Common Interview Questions

❓ Q1: L4 vs L7 কখন কোনটা use করবে?
উত্তর: L4 ব্যবহার করো — ultra-low latency দরকার হলে, non-HTTP protocol (game server, real-time chat), simple TCP traffic। L7 ব্যবহার করো — path-based routing, SSL termination, rate limiting, microservices। সাধারণত web app এ L7 recommended।
❓ Q2: Sticky Session কী এবং কেন problem?
উত্তর: Sticky session (IP Hash) মানে same user সবসময় same server এ যায়। Problem: সেই server down হলে user এর session হারায়। Better solution: Session data Redis এ store করো — তাহলে যেকোনো server session access করতে পারবে।
❓ Q3: Load Balancer নিজে single point of failure না?
উত্তর: হ্যাঁ এটা একটা valid concern। Solution: Active-Passive (একটা primary, একটা standby) বা Active-Active (দুটোই simultaneously serve করে) LB setup। Cloud providers (AWS ALB) automatically highly available হয়।
