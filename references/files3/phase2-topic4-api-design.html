01 — Core Concept

API কী এবং কেন Design Matter করে?

API (Application Programming Interface) হলো দুটো system এর মধ্যে communication এর চুক্তি। ধরো একটা restaurant এর menu — তুমি menu দেখে order করো, kitchen কীভাবে বানায় সেটা তোমাকে জানতে হয় না। API ও ঠিক তাই — client কী চাইতে পারে এবং server কী দেবে সেটা define করে।

💡 API Design কেন গুরুত্বপূর্ণ
ভালো API design মানে developer-friendly, predictable, consistent। খারাপ API মানে confusion, bugs, maintenance nightmare। Twitter এর API v1 থেকে v2 migration developers দের কাঁদিয়েছে। Good design upfront সেই pain বাঁচায়।
02 — REST API

REST — Representational State Transfer

REST হলো সবচেয়ে popular API style। HTTP methods ব্যবহার করে resources manipulate করে। Stateless, cacheable, uniform interface — এই principles এ কাজ করে।

REST Best Practices

✅ REST Design Rules
Nouns use করো, verbs না: /users ভালো, /getUsers খারাপ। Plural use করো: /users, /products। Nesting: /users/123/orders। Versioning: /api/v1/users। HTTP status codes সঠিকভাবে use করো।

03 — GraphQL

GraphQL — Query Language for APIs

Facebook তৈরি করেছে। Client exactly যেটুকু data চায় সেটুকুই request করতে পারে — over-fetching বা under-fetching নেই। একটা endpoint, flexible queries।

❌ REST Problem

User profile page এ name, avatar দরকার। GET /users/123 → পুরো object আসে (email, phone, address সব)। Over-fetching! আবার, posts ও দরকার হলে আরেকটা request। Under-fetching!

✅ GraphQL Solution

একটা query তে exactly যা দরকার: { user(id: 123) { name avatar posts { title } } } — শুধু name, avatar, এবং post titles আসবে। একটা request এ সব।

GraphQLquery.graphql
# REST এ এটার জন্য 3টা API call লাগতো
# GraphQL এ একটাই
query GetUserDashboard {
  user(id: "123") {
    name
    avatar
    posts(first: 5) {
      title
      createdAt
    }
    followers {
      count
    }
  }
}

# Mutation — data change করতে
mutation UpdateProfile($name: String!, $bio: String) {
  updateUser(input: { name: $name, bio: $bio }) {
    id
    name
    updatedAt
  }
}
⚠️ GraphQL এর Downside
Caching complex (REST এর মতো URL-based cache কাজ করে না)। N+1 query problem (DataLoader দিয়ে solve করতে হয়)। Simple CRUD apps এ overkill। REST এর মতো HTTP cache/CDN সুবিধা নেই।
04 — gRPC

gRPC — High Performance RPC

Google তৈরি করেছে। Protocol Buffers (protobuf) দিয়ে data serialize করে — JSON এর চেয়ে ৫-১০x faster। Strongly typed, code generation, bidirectional streaming। Microservices internal communication এর জন্য ideal।

Protobufuser.proto
// Service definition
syntax = "proto3";

service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
  rpc ListUsers (ListRequest) returns (stream UserResponse);
}

message UserRequest {
  string user_id = 1;
}

message UserResponse {
  string id = 1;
  string name = 2;
  string email = 3;
  int32  age = 4;
}

05 — API Gateway

API Gateway — Single Entry Point

Microservices architecture এ ৫০টা service আছে। Client কি ৫০টা আলাদা URL জানবে? না। API Gateway হলো single entry point — সব requests এখান দিয়ে ঢোকে, Gateway সঠিক service এ route করে।

✅ API Gateway Features
Authentication: JWT token verify করে, backend services কে auth চিন্তা করতে হয় না। Rate Limiting: প্রতি user/IP কতটা request করতে পারবে limit করে। Request Transformation: Request/response format modify করতে পারে। Circuit Breaking: Backend service fail হলে fast-fail করে। Popular tools: AWS API Gateway, Kong, Nginx, Traefik।
06 — Code Examples

Code Examples

Node.js — REST API with Express

Node.jsrest_api.js
const express = require('express');
const app = express();
app.use(express.json());

// GET /api/v1/users/:id
app.get('/api/v1/users/:id', async (req, res) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) return res.status(404).json({
      error: 'User not found', code: 'USER_NOT_FOUND'
    });
    res.status(200).json({ data: user });
  } catch (err) {
    res.status(500).json({ error: 'Internal server error' });
  }
});

// POST /api/v1/users — create user
app.post('/api/v1/users', async (req, res) => {
  const { name, email, password } = req.body;
  if (!name || !email) return res.status(400).json({
    error: 'name and email required'
  });
  const user = await User.create({ name, email, password });
  res.status(201).json({ data: user });
});

// PATCH /api/v1/users/:id — partial update
app.patch('/api/v1/users/:id', async (req, res) => {
  const updated = await User.findByIdAndUpdate(
    req.params.id, req.body, { new: true }
  );
  res.json({ data: updated });
});

Rate Limiting Middleware

Node.jsrateLimit.js
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');

// API Rate limit: 100 requests per 15 minutes per IP
const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 100,
  standardHeaders: true,
  store: new RedisStore({ client: redisClient }),
  handler: (req, res) => {
    res.status(429).json({
      error: 'Too many requests',
      retryAfter: Math.ceil(res.getHeader('Retry-After'))
    });
  }
});

app.use('/api/', apiLimiter);
07 — Interview Prep

Common Interview Questions

❓ Q1: REST vs GraphQL — কোনটা better?
উত্তর: "Better" নির্ভর করে use case এ। REST — simple CRUD, public API, caching important। GraphQL — complex data requirements, mobile app (bandwidth sensitive), multiple clients different data চাই। Interview এ দুটোর trade-off বলো।
❓ Q2: API Versioning কীভাবে করবে?
উত্তর: ৩টা approach: URL versioning (/api/v1/ → সবচেয়ে common, visible)। Header versioning (Accept: application/vnd.api+json;version=1 → cleaner URLs)। Query param (?version=1 → easy but messy)। URL versioning সবচেয়ে popular কারণ explicit এবং debuggable।
❓ Q3: gRPC কখন use করবে?
উত্তর: Microservices internal communication এ — high throughput, low latency দরকার। Binary protocol বলে JSON এর চেয়ে faster। Strongly typed — contract clear। Browser থেকে directly call কঠিন বলে public API তে REST/GraphQL ভালো।

MCQ Quiz — API Design

Score: 0/10
Question 01 / 10
REST API তে নতুন resource create করতে কোন HTTP method use করবে?
A GET
B POST
C PUT
D PATCH
সঠিক উত্তর: B — POST নতুন resource create করে এবং 201 Created return করে। PUT পুরো resource replace করে (existing হলে)। PATCH partial update।
Question 02 / 10
GraphQL এর সবচেয়ে বড় advantage REST এর উপর কী?
A Faster than REST
B Better security
C Client exactly যেটুকু data চায় সেটুকুই request করতে পারে — no over/under-fetching
D Easier to implement
সঠিক উত্তর: C — GraphQL এর core benefit হলো flexible queries। Client ঠিক করে কোন fields দরকার। এতে over-fetching (extra data) এবং under-fetching (multiple requests) দুটোই avoid হয়।
Question 03 / 10
HTTP 401 vs 403 — পার্থক্য কী?
A দুটো একই — authentication error
B 401 server error, 403 client error
C 401 more severe than 403
D 401 = unauthenticated (login করোনি), 403 = unauthorized (login করেছো কিন্তু permission নেই)
সঠিক উত্তর: D — 401: Token নেই বা invalid। "Who are you?" 403: Token valid কিন্তু এই resource access করার permission নেই। "I know you, but you can't enter here."
Question 04 / 10
API Gateway এর primary কাজ কী?
A Single entry point হিসেবে কাজ করে, auth, rate limiting, routing করে
B Database connection manage করে
C Frontend code serve করে
D Message queue manage করে
সঠিক উত্তর: A — API Gateway microservices architecture এর front door। সব requests এখান দিয়ে যায়। Auth verify করে, rate limit check করে, সঠিক service এ route করে।
Question 05 / 10
gRPC কোন data format ব্যবহার করে?
A JSON
B Protocol Buffers (Protobuf) — Binary
C XML
D YAML
সঠিক উত্তর: B — gRPC Protocol Buffers (protobuf) ব্যবহার করে। Binary format বলে JSON এর চেয়ে ৫-১০x smaller এবং faster serialize/deserialize হয়। Strongly typed schema।
Question 06 / 10
REST API versioning এর সবচেয়ে popular approach কোনটা?
A Query parameter: /users?version=2
B Header: Accept-Version: v2
C URL path: /api/v2/users
D Subdomain: v2.api.example.com
সঠিক উত্তর: C — URL path versioning (/api/v1/, /api/v2/) সবচেয়ে common। Explicit, visible, easy to debug, browser friendly। Twitter, Stripe, GitHub সবাই এটা use করে।
Question 07 / 10
REST এ /getUsers endpoint বানানো কেন ভালো practice নয়?
A Get method এ capital letter ব্যবহার করা যায় না
B URL এ underscore ব্যবহার করা যায় না
C Too long URL
D REST এ URL এ verbs নয়, nouns use করা উচিত। সঠিক: GET /users
সঠিক উত্তর: D — REST best practice হলো URL এ nouns (resources): /users, /products, /orders। Action/verb টা HTTP method দিয়ে express করো: GET (read), POST (create), DELETE (remove)।
Question 08 / 10
Rate Limiting কী করে?
A Per user/IP request limit করে — abuse এবং DDoS থেকে protect করে
B API response slow করে দেয়
C Database query কমায়
D Large responses reject করে
সঠিক উত্তর: A — Rate Limiting প্রতি user বা IP কে নির্দিষ্ট সময়ে নির্দিষ্ট requests করতে দেয়। Excess হলে 429 Too Many Requests। Abuse prevention, DDoS protection, fair usage নিশ্চিত করে।
Question 09 / 10
Mobile app এর জন্য কোন API style সবচেয়ে efficient?
A REST — সবচেয়ে popular
B gRPC — fastest
C GraphQL — client exactly needed data request করতে পারে, bandwidth save হয়
D SOAP — most reliable
সঠিক উত্তর: C — Mobile এ bandwidth valuable। GraphQL এ client শুধু দরকারী fields request করে — over-fetching নেই। আর একটা query তে nested data পাওয়া যায় — multiple round trips নেই।
Question 10 / 10
একটা fintech startup এর internal microservices communication এর জন্য কোনটা সবচেয়ে suitable?
A REST — সবচেয়ে familiar
B gRPC — high performance, strongly typed, internal services এর জন্য ideal
C GraphQL — flexible queries
D WebSocket — real-time
সঠিক উত্তর: B — Internal microservices communication এ gRPC excellent: binary protocol (fast), strongly typed contract (safe), HTTP/2 (multiplexing), code generation (less boilerplate)। Browser থেকে call করতে হয় না বলে browser limitation relevant না।
