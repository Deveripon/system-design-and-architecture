01 — Core Concept

Message Queue কী?

ধরো Swiggy তে order দিলে — app বলে "Order Confirmed!" এবং তুমি phone রেখে দিলে। পরে notification আসলো, restaurant prepare করলো, rider আসলো। তুমি order দেওয়ার পরেই চলে যেতে পারলে — wait করতে হলো না। এটাই asynchronous messaging। Synchronous হলে তুমি phone ধরে থাকতে হতো যতক্ষণ food deliver না হয়!

💡 Definition
Message Queue হলো একটি middleware যেখানে Producer messages পাঠায় এবং Consumer সেগুলো নিজের pace এ process করে। দুটো service সরাসরি কথা বলে না — Queue মাঝখানে buffer হিসেবে থাকে। এতে services decoupled থাকে এবং system resilient হয়।
02 — Why Async?

কেন Async Messaging দরকার?

⚡ Decoupling

Service A এবং B একে অপরকে directly depend করে না। B down হলেও A কাজ করে, messages queue এ জমা থাকে।

📈 Traffic Spike

হঠাৎ ১০,০০০ orders আসলে queue এ জমা হয়। Consumer নিজের pace এ process করে। System crash হয় না।

🔄 Retry Logic

Processing fail হলে message queue এ ফিরে আসে এবং retry হয়। Data loss হয় না।

🏗️ Microservices

Services loosely coupled থাকে। একটা service update করলে অন্যটা affect হয় না।

📊 Real Example — E-commerce Order Flow
Payment done → Queue তে event পাঠাও: "order_placed". তারপর parallel process: Email Service (confirmation পাঠাও), SMS Service (notify করো), Inventory Service (stock কমাও), Analytics Service (track করো)। User ২ seconds এ response পায়, বাকি সব background এ হয়।
03 — MQ Patterns

Message Queue Patterns

1. Point-to-Point (Queue Model)

একটা message শুধু একটাই consumer পায়। Task queue এর জন্য perfect — একটা job একবারই process হওয়া উচিত।

2. Publish-Subscribe (Topic/Fan-out)

একটা message multiple consumers পায়। Event broadcast এর জন্য — "order_placed" event পেলে সবাই process করে।

04 — Kafka Deep Dive

Apache Kafka — Event Streaming Platform

Kafka শুধু message queue না — এটা distributed event streaming platform। LinkedIn তৈরি করেছিল। এখন LinkedIn, Uber, Netflix, Airbnb সবাই use করে। প্রতিদিন trillions of events handle করে।

🔑 Kafka Key Terms
Topic: Message category (e.g. "order-events")। Partition: Topic কে parallel শাখায় ভাগ করা। Offset: Partition এ message এর sequential ID। Producer: Message পাঠায়। Consumer Group: Messages consume করে। Broker: Kafka server node। Replication Factor: কতটা copy রাখবে।
✅ Kafka এর Unique Power
Messages replay করা যায় (offset থেকে)। Multiple Consumer Groups একই Topic থেকে independently read করতে পারে। Email Service, Analytics Service, Inventory Service — তিনজনই same event পাবে কিন্তু আলাদা আলাদা। এটাই Kafka কে "event log" বানায়, শুধু queue না।
05 — RabbitMQ

RabbitMQ — Traditional Message Broker

RabbitMQ AMQP protocol ব্যবহার করে। Complex routing, acknowledgment, dead letter queue — এসব feature এর জন্য। Push-based: broker নিজেই consumer কে message push করে।

🐰 RabbitMQ Key Concepts
Exchange: Message receive করে Queue তে route করে। Types: Direct, Topic, Fanout, Headers। Queue: Messages store করে। Binding: Exchange-Queue connection। ACK: Consumer সফলভাবে process করলে acknowledge করে। NACK + DLQ: Fail হলে Dead Letter Queue তে যায়।

Python — Kafka Producer & Consumer

Pythonkafka_example.py
from kafka import KafkaProducer, KafkaConsumer
import json, time

# ── Producer — Order event পাঠাও ───────────────
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode()
)

def place_order(order_data):
    event = {
        'event_type': 'order_placed',
        'order_id': order_data['id'],
        'user_id': order_data['user_id'],
        'amount': order_data['amount'],
        'timestamp': time.time()
    }
    # Topic: 'order-events' এ publish করো
    future = producer.send('order-events', event)
    result = future.get(timeout=10)
    print(ff"✅ Event published → partition:{result.partition} offset:{result.offset}")

# ── Consumer: Email Service ─────────────────────
email_consumer = KafkaConsumer(
    'order-events',
    bootstrap_servers=['localhost:9092'],
    group_id='email-service',    # Consumer group name
    value_deserializer=lambda m: json.loads(m.decode()),
    auto_offset_reset='earliest'  # Start from beginning
)

print("📧 Email service listening...")
for message in email_consumer:
    event = message.value
    if event['event_type'] == 'order_placed':
        print(ff"📧 Sending email for order: {event['order_id']}")
        # send_confirmation_email(event['user_id'])

# NOTE: Analytics service can ALSO consume same topic
# with group_id='analytics-service' — independently!

Node.js — RabbitMQ Task Queue

Node.jsrabbitmq_worker.js
const amqp = require('amqplib');

// ── Publisher: Task পাঠাও ──────────────────────
async function publishImageTask(imageUrl) {
  const conn = await amqp.connect('amqp://localhost');
  const ch = await conn.createChannel();

  await ch.assertQueue('image_resize', { durable: true });

  ch.sendToQueue(
    'image_resize',
    Buffer.from(JSON.stringify({ imageUrl, size: '800x600' })),
    { persistent: true }  // Survive broker restart
  );
  console.log(`📤 Task queued: ${imageUrl}`);
  await conn.close();
}

// ── Worker: Task process করো ───────────────────
async function startWorker() {
  const conn = await amqp.connect('amqp://localhost');
  const ch = await conn.createChannel();

  await ch.assertQueue('image_resize', { durable: true });
  ch.prefetch(1);  // Process one task at a time

  console.log('🔧 Worker ready, waiting for tasks...');
  ch.consume('image_resize', async (msg) => {
    const task = JSON.parse(msg.content.toString());
    console.log(`🖼️ Resizing: ${task.imageUrl}`);

    try {
      await resizeImage(task.imageUrl, task.size);
      ch.ack(msg);             // ✅ Success: remove from queue
    } catch (err) {
      console.error('❌ Failed, requeuing...', err);
      ch.nack(msg, false, true); // ❌ Fail: requeue message
    }
  });
}

startWorker();

// Dead Letter Queue — failed message গুলো আলাদা queue তে যাবে
await ch.assertExchange('dlx', 'direct');
await ch.assertQueue('failed_tasks');
await ch.bindQueue('failed_tasks', 'dlx', 'failed');

// Main queue এ DLX set করো
await ch.assertQueue('image_resize', {
  durable: true,
  arguments: {
    'x-dead-letter-exchange': 'dlx',
    'x-dead-letter-routing-key': 'failed',
    'x-message-ttl': 30000  // 30s TTL, then DLQ
  }
});
// এখন failed/expired messages → 'failed_tasks' queue এ যাবে

🎯 Quick Decision Guide
Kafka বেছে নাও: Real-time data pipeline, event sourcing, multiple consumers same data চাই, replay দরকার।
RabbitMQ বেছে নাও: Complex routing, task queue, explicit ACK/NACK, Dead Letter Queue দরকার।
AWS SQS বেছে নাও: AWS stack এ আছো, managed solution চাও, overhead কমাতে চাও।
08 — Interview Prep

Common Interview Questions

❓ Q1: Kafka এ at-least-once vs exactly-once delivery কী?
উত্তর: At-least-once: message কমপক্ষে একবার deliver হবে, কিন্তু duplicate possible। Exactly-once: Kafka transactions ব্যবহার করে duplicate ছাড়া deliver। সাধারণত at-least-once + idempotent consumer (same message দুবার process করলেও একই result) বেশি practical।
❓ Q2: Kafka Partition কীভাবে throughput বাড়ায়?
উত্তর: Partition parallel processing enable করে। ৩টা partition থাকলে ৩টা consumer simultaneously আলাদা partition থেকে read করতে পারে। Partition = parallelism। বেশি partition = বেশি throughput, কিন্তু overhead বাড়ে।
❓ Q3: Message Queue vs Direct API call — কখন কোনটা?
উত্তর: Direct API call — যখন immediate response দরকার (login, payment confirmation)। Message Queue — যখন background processing চলবে (email, notification, analytics), services decoupled রাখতে চাই, traffic spike handle করতে হবে।
❓ Q4: Dead Letter Queue কী কাজ করে?
উত্তর: যে messages process করতে বারবার fail হচ্ছে (max retry exceed করেছে) সেগুলো DLQ তে যায়। মূল queue block হয় না। Later DLQ inspect করে manually fix করা যায় বা alert trigger করা যায়।

Dead Letter Queue (DLQ) কেন ব্যবহার করা হয়?

Question 07 / 10
Kafka vs RabbitMQ — কোন scenario তে Kafka বেশি suitable?

D Simple order confirmation
সঠিক উত্তর: A — Kafka high-throughput streaming এর জন্য designed। Millions/sec events, multiple consumer groups, replay — এই সব Kafka এর strength। Image resize এর মতো low-volume task queue এ RabbitMQ simpler।
Question 08 / 10
Kafka Consumer Group এর কাজ কী?

B Messages compress করে

সঠিক উত্তর: D — Consumer Group এ একই group এর consumers partitions ভাগ করে নেয়। ৩ partition + ৩ consumers → প্রতিজন একটা partition। এতে parallel processing হয়। Different groups same topic independently consume করতে পারে।
