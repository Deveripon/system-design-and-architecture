01 — Core Concept

Cache কী এবং কেন দরকার?

ধরো তুমি প্রতিদিন office যাও। বাসা থেকে office যেতে ৩০ মিনিট লাগে। কিন্তু তুমি যদি office এর কাছেই থাকো — তাহলে ৫ মিনিটেই পৌঁছাও। Cache হলো সেই কাছের বাসা — frequently used data কে database এর বদলে memory তে রেখে দ্রুত serve করা।

💡 Definition
Cache হলো একটি high-speed data storage layer যেখানে frequently accessed data temporarily store করা হয়, যাতে পরেরবার ওই data দরকার হলে original (slow) source এর বদলে cache থেকে পাওয়া যায়। এটি database query কমায়, latency কমায়, এবং throughput বাড়ায়।

Cache ছাড়া কী হয়?

ধরো একটা news website এ "Top 10 News" section আছে। প্রতি second ১০,০০০ user এই page visit করছে। Cache ছাড়া প্রতিটা request এ database query করতে হবে — যার মানে ১০,০০০ × database query per second! Database crash করবে, site slow হবে, user চলে যাবে।

02 — Why It Matters

কেন Cache জানা দরকার?

⚡ Performance

Database থেকে data fetch করতে ১০০-৫০০ms লাগে। Cache থেকে লাগে ১ms এরও কম। ১০০x-৫০০x speedup পাওয়া সম্ভব।

💰 Cost Saving

Database server expensive। Cache ব্যবহার করলে DB queries কমে, server load কমে, hosting cost কমে। Facebook cache দিয়ে প্রতি বছর কোটি টাকা বাঁচায়।

📈 Scalability

Database horizontally scale করা কঠিন। কিন্তু cache layer add করলে DB এর উপর load না বাড়িয়ে traffic handle করা যায়।

🛡️ Availability

DB temporarily down হলেও cached data serve করা যায়। Users কিছুটা হলেও response পাবে — complete outage হবে না।

📊 Real Numbers
Twitter প্রতি second ৬,০০০+ tweets serve করে। Cache ছাড়া এটা impossible। Amazon এর research বলে — প্রতি ১০০ms latency বাড়লে sales ১% কমে। Google ৫০০ms slow হলে ২০% কম searches হয়।
03 — How It Works

Cache কীভাবে কাজ করে?

Cache Hit vs Cache Miss

Cache এর সবচেয়ে important concept হলো Cache Hit এবং Cache Miss।

Cache Eviction Policies

Cache এর memory সীমিত। পুরানো data বাদ দেওয়ার জন্য আলাদা strategy আছে:

PolicyFull Nameকীভাবে কাজ করেকখন ব্যবহার
LRU Least Recently Used সবচেয়ে পুরানো ব্যবহার করা item বাদ দেয় সবচেয়ে common, default choice
LFU Least Frequently Used সবচেয়ে কম ব্যবহার হওয়া item বাদ দেয় Hot items keep করতে

04 — Cache Strategies

৪টি প্রধান Caching Pattern

1. Cache-Aside (Lazy Loading)

Application নিজেই cache manage করে। Data দরকার হলে আগে cache check করে, না পেলে DB থেকে আনে এবং cache এ রাখে। সবচেয়ে popular pattern।

✅ Cache-Aside এর সুবিধা
Cache failure হলেও app কাজ করে (DB fallback আছে)। Only requested data cache হয়। Read-heavy apps এর জন্য perfect।

2. Write-Through

Data write করার সময় একসাথে Cache এবং DB — দুটোতেই লেখা হয়। Cache সবসময় fresh থাকে।

3. Write-Behind (Write-Back)

আগে শুধু Cache এ লেখা হয়, পরে asynchronously DB তে sync হয়। খুব fast কিন্তু cache crash হলে data loss এর risk আছে।

4. Read-Through

Cache নিজেই DB থেকে data load করে। App শুধু Cache কে জিজ্ঞেস করে — Cache Miss হলে Cache নিজেই DB তে যায়।

PatternRead SpeedWrite SpeedConsistencyBest For
Cache-AsideFastNormalEventualRead-heavy apps
Write-ThroughFastSlowStrongWrite consistency needed
Write-BehindFastVery FastWeakHigh-write, loss-tolerant
Read-ThroughFastNormalEventualSimplified app logic
05 — Redis Deep Dive

Redis — এর ভেতরে কী আছে?

Redis (Remote Dictionary Server) হলো সবচেয়ে popular in-memory data store। এটা শুধু cache না — এটা data structure server। Redis ব্যবহার করে Twitter, GitHub, Stack Overflow, Airbnb সহ হাজারো company।

🔴 Redis এর বিশেষত্ব
Redis single-threaded হলেও প্রতি second ১ million+ operations handle করতে পারে কারণ সব data RAM এ থাকে। I/O operations asynchronous। Simple operations O(1) time complexity।

Redis Data Structures

Redis শুধু key-value store না — এটা ৫টি আলাদা data structure support করে:

TypeCommand ExampleUse Case
StringSET user:1 "John"Session token, counter, simple cache
HashHSET user:1 name "John" age 25User profile, object store
ListLPUSH feed:1 "tweet123"Activity feed, message queue, log
SetSADD tags:post1 "redis"Unique visitors, tagging, friends list
Sorted SetZADD leaderboard 100 "user1"Leaderboard, rate limiting, timeline

Redis Persistence

Redis in-memory হলেও data হারানো যাবে না — তাই দুটো persistence option আছে:

RDB (Snapshot)

নির্দিষ্ট interval এ পুরো dataset এর snapshot নেয়। Fast restart, কিন্তু শেষ snapshot এর পর এর data হারাতে পারে।

AOF (Append Only File)

প্রতিটা write operation log করে। Near-zero data loss, কিন্তু file বড় হয়, restart একটু slow।

06 — Code Examples

হাতে-কলমে Code

Python — Basic Redis Operations

Pythonredis_basics.py
import redis
import json

# Redis connection
r = redis.Redis(host='localhost', port=6379, decode_responses=True)

# ── String Operations ──────────────────────────
r.set('user:1:name', 'Rahim Ahmed')
r.set('user:1:name', 'Rahim Ahmed', ex=3600)  # TTL: 1 hour

name = r.get('user:1:name')
print(name)  # Rahim Ahmed

# ── Hash Operations (Object Store) ─────────────
r.hset('user:1', mapping={
    'name': 'Rahim Ahmed',
    'email': 'rahim@example.com',
    'age': '28'
})

user = r.hgetall('user:1')
print(user)  # {'name': 'Rahim Ahmed', 'email': ..., 'age': '28'}

# ── Counter (views, likes) ─────────────────────
r.incr('post:42:views')   # increment by 1
r.incrby('post:42:views', 10)
views = r.get('post:42:views')

# ── Sorted Set — Leaderboard ───────────────────
r.zadd('game:leaderboard', {
    'player1': 5000,
    'player2': 7500,
    'player3': 3200
})

# Top 3 players
top3 = r.zrevrange('game:leaderboard', 0, 2, withscores=True)
print(top3)  # [('player2', 7500.0), ('player1', 5000.0), ...]

Node.js — Cache-Aside Pattern (Real World)

Node.jscacheService.js
const redis = require('redis');
const { Pool } = require('pg');  // PostgreSQL

const client = redis.createClient({ url: 'redis://localhost:6379' });
const db = new Pool({ connectionString: process.env.DATABASE_URL });

// Cache-Aside Pattern implementation
async function getUserById(userId) {
  const cacheKey = `user:${userId}`;

  // Step 1: Cache check করো
  const cached = await client.get(cacheKey);
  if (cached) {
    console.log('✅ Cache HIT');
    return JSON.parse(cached);
  }

  // Step 2: Cache Miss → DB query
  console.log('❌ Cache MISS → Querying DB');
  const result = await db.query(
    'SELECT * FROM users WHERE id = $1', [userId]
  );
  const user = result.rows[0];

  // Step 3: Cache এ store করো (TTL: 1 hour)
  if (user) {
    await client.setEx(cacheKey, 3600, JSON.stringify(user));
  }

  return user;
}

// Cache Invalidation — data update হলে cache মুছো
async function updateUser(userId, data) {
  await db.query(
    'UPDATE users SET name=$1, email=$2 WHERE id=$3',
    [data.name, data.email, userId]
  );
  // Cache invalidate করো
  await client.del(`user:${userId}`);
  console.log('🗑️ Cache invalidated for user:', userId);
}

Rate Limiting with Redis

Node.jsrateLimiter.js
// API Rate Limiting: 100 requests per minute per user
async function checkRateLimit(userId) {
  const key = `rate:${userId}:${Math.floor(Date.now() / 60000)}`;

  const pipeline = client.multi();
  pipeline.incr(key);
  pipeline.expire(key, 60);  // 60 seconds TTL

  const [count] = await pipeline.exec();

  if (count > 100) {
    throw new Error('Rate limit exceeded. Try again in 1 minute.');
  }

  return { allowed: true, remaining: 100 - count };
}
07 — Real World

Real World Use Cases

🐦 Twitter

User timeline cache করে Redis এ। @elonmusk এর tweet ১ বিলিয়ন মানুষ দেখে — প্রতিবার DB query impossible। Redis cluster ব্যবহার করে।

🛒 Amazon

Product details, price, inventory — সব cache করা। Shopping cart টা Redis এ রাখে। Session management এর জন্য।

🎮 Gaming

Leaderboard Redis Sorted Sets দিয়ে। Real-time score update, top players instantly দেখা — Redis ZSet এর perfect use case।

🔐 Auth Systems

JWT token blacklist, OTP store, session data — সব Redis এ। TTL দিয়ে auto-expire। Token revoke করতে Redis DEL command।

⚠️ Cache Stampede Problem
একটা popular key expire হলে হঠাৎ হাজারো request একসাথে DB তে যায় — এটাকে Cache Stampede বা Thundering Herd বলে। Solution: Mutex Lock (শুধু একটা request DB তে যাবে, বাকিরা wait করবে) অথবা Probabilistic early expiration (expire এর একটু আগেই refresh করা)।
08 — Interview Prep

Common Interview Questions

❓ Q1: Cache Invalidation কীভাবে করবে?
উত্তর: ৩টি approach: (১) TTL — নির্দিষ্ট সময় পর auto-expire। (২) Event-driven — data update হলে cache.del() করো। (৩) Write-Through — সবসময় DB এবং Cache একসাথে update করো। বলো — "এটা system এর consistency requirement এর উপর নির্ভর করে।"
❓ Q2: Cache Hit Rate কীভাবে বাড়াবে?
উত্তর: (১) Cache size বাড়াও (২) Eviction policy সঠিক choose করো (LRU সাধারণত ভালো) (৩) TTL carefully set করো — খুব কম হলে miss বেশি, খুব বেশি হলে stale data। (৪) Hot data identify করে আগেই warm-up করো।
❓ Q3: Redis single-threaded হয়েও কীভাবে এত fast?
উত্তর: সব data RAM এ থাকে (disk I/O নেই)। Event loop দিয়ে async I/O। Simple data structures (O(1) operations)। Network overhead minimize করতে pipelining। CPU bottleneck কম কারণ operations খুব simple।
❓ Q4: Cache কোথায় রাখবে — Client-side নাকি Server-side?
উত্তর: Client-side (browser cache, CDN) — static assets এর জন্য। Server-side (Redis, Memcached) — dynamic data, session, computed results। উভয়ই হতে পারে — multi-layer caching strategy।

Redis দিয়ে একটা Simple News Feed Cache System বানাও।

News Feed Caching with Redis

একটা Express.js API বানাবো যেটা news articles কে Redis এ cache করে। DB হিসেবে simple JSON file use করবো।

1

Setup করো

npm install express redis — তারপর Redis start করো: redis-server

2

Fake DB বানাও

articles.json file এ ১০টা fake news article রাখো (id, title, content, author)।

3

Cache-Aside API implement করো

GET /articles/:id route — আগে Redis check, miss হলে JSON file থেকে load, তারপর Redis এ store।

4

Response time measure করো

First request (cache miss) এবং second request (cache hit) এর time compare করো। Cache hit কত times faster?

5

Cache Invalidation add করো

PUT /articles/:id route — update করলে cache invalidate হবে।

6

Bonus: TTL add করো

News articles ১ ঘণ্টা পর expire হবে। TTL track করে response header এ পাঠাও।
